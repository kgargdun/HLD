## **6.1 Consistent Hashing (Deep Dive)**

> Used when **state or cache affinity matters**

---

### Problem with Normal Hashing

```id="u8d6y1"
node = hash(key) % N
```

* Adding/removing nodes → massive reshuffling ❌

---

### Consistent Hashing Idea

* Hash nodes + keys on a **ring**
* Key assigned to nearest node clockwise

---

### Benefits

* Minimal key movement
* Stable distribution

---

### Virtual Nodes

* Each physical node mapped multiple times

**Why?**

* Avoid uneven distribution

---

**Where used:**

* Distributed caches (e.g., Redis clusters)
* CDN routing

---

**Interview Line:**

> “I’ll use consistent hashing when I need stable routing across dynamic nodes.”

## **6.2 Global Load Balancing**

### Problem

* Users across the world
* Need low latency

---

### Solution

#### Geo-based Routing

* Route to nearest region

```id="klb3gj"
User (India) → India DC  
User (US) → US DC
```

---

#### Latency-based Routing

* Choose fastest responding region

---

#### Health-aware Routing

* Avoid unhealthy regions

---

**Tools:**

* AWS Route 53

---

**Tradeoff:**

* Accuracy vs complexity

---

## **6.3 Anycast (Advanced Networking)**

### Idea

* Same IP advertised from multiple locations

---

### How it works

* Network routes request to nearest node

---

### Benefits

* Low latency
* Automatic failover

---

### Used by:

* CDNs
* DNS systems

---

**Interview Line:**

> “Anycast provides network-level load balancing with fast failover.”
