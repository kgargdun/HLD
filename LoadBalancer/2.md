## **Load Balancing Algorithms.md**

---

## **2.1 Round Robin**

**Idea:**

* Requests distributed sequentially across servers

```
Req1 → S1  
Req2 → S2  
Req3 → S3  
Req4 → S1 ...
```

---

### Pros

* Simple
* No state needed

---

### Cons

* Ignores server capacity
* Ignores load differences

---

### When to Use

* Identical servers
* Uniform workloads

---

## **2.2 Weighted Round Robin**

**Idea:**

* Servers assigned weights based on capacity

```
S1 (weight 2), S2 (weight 1)  
→ S1 gets 2x traffic
```

---

### Pros

* Handles heterogeneous servers

---

### Cons

* Static weights → doesn’t adapt dynamically

---

## **2.3 Least Connections**

**Idea:**

* Route request to server with fewest active connections

---

### Pros

* Adapts to real-time load
* Works well for long-lived connections

---

### Cons

* Slightly more complex
* Needs tracking state

---

### Use Case

* APIs with variable processing time

---

## **2.4 Least Response Time**

**Idea:**

* Choose server with lowest latency

---

### Pros

* Optimizes user experience
* Adapts to slow nodes

---

### Cons

* Requires continuous monitoring
* Noisy metric (fluctuations)

---

## **2.5 IP Hash**

**Idea:**

* Hash(client IP) → server

---

### Pros

* Same user → same server
* Useful for session stickiness

---

### Cons

* Uneven distribution
* Problems if nodes change

---

## **2.6 Consistent Hashing (IMPORTANT)**

**Idea:**

* Hash servers + keys on a ring
* Minimal remapping when nodes change

---

### Pros

* Stable distribution
* Great for caching + stateful systems

---

### Cons

* More complex
* Needs virtual nodes for balance

---

## **Comparison Summary**

| Algorithm           | Adaptiveness | Complexity | Use Case            |
| ------------------- | ------------ | ---------- | ------------------- |
| Round Robin         | Low          | Low        | Simple systems      |
| Weighted RR         | Low          | Low        | Unequal servers     |
| Least Connections   | Medium       | Medium     | Variable load       |
| Least Response Time | High         | High       | Latency-sensitive   |
| IP Hash             | Low          | Low        | Sticky sessions     |
| Consistent Hashing  | Medium       | High       | Distributed systems |

---

## **Interview Playbook (Algorithms)**

Default answer:

> “I’ll start with least connections since it adapts to real-time load.”

Then refine:

* Sticky sessions needed → IP hash
* Distributed system → consistent hashing
* Simple system → round robin

---

## **Common Mistakes**

* Always saying “round robin”
* Ignoring dynamic load
* Not considering session stickiness

---

## **Golden Rule**

> “Choose algorithm based on workload behavior, not simplicity.”
