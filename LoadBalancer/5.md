## **5.1 Industry Tools & Systems**

### NGINX

**Type:** L7 Load Balancer

**Features:**

* Reverse proxy
* SSL termination
* Path-based routing
* Rate limiting

**Used for:**

* API gateways
* Web traffic routing

---

### HAProxy

**Type:** L4 + L7

**Features:**

* Very high performance
* Fine-grained control
* Advanced routing

**Used for:**

* High-throughput systems
* Low-latency services

---

### AWS Load Balancers

#### ALB (Application Load Balancer)

* L7
* HTTP/HTTPS routing

#### NLB (Network Load Balancer)

* L4
* Ultra-low latency

#### ELB (Classic)

* Older generation

---

**Interview Line:**

> “In production, we usually use managed load balancers like AWS ALB or NGINX depending on control vs convenience.”

## **5.3 Decision Framework (Very Important)**

### Step 1: Traffic Type

* HTTP → L7
* TCP/UDP → L4

---

### Step 2: Routing Needs

* Simple → Round robin
* Dynamic → Least connections
* Sticky → IP hash

---

### Step 3: Scale

* Small → Single LB
* Large → Distributed LB

---

### Step 4: Availability

* Multi-LB setup
* Health checks

---

### Step 5: Deployment

* Managed (AWS) vs Self-hosted (NGINX)

---

## **5.4 Tradeoffs (Core Interview Content)**

### Performance vs Flexibility

* L4 → fast but limited
* L7 → flexible but slower

---

### Cost vs Control

* Managed LB → easy but expensive
* Self-hosted → flexible but operational overhead

---

### Stateless vs Stateful

* Stateless → scalable
* Stateful (sticky sessions) → simpler but risky

---

### Simplicity vs Scalability

* Single LB → simple
* Distributed LB → scalable but complex

---

**Interview Line:**

> “Choosing LB involves balancing performance, flexibility, and operational complexity.”

## **5.7 Interview Answer Templates**

---

### Q: “How would you add load balancing?”

**Answer:**

1. “Introduce a load balancer in front of app servers”
2. “Use least connections for better distribution”
3. “Add health checks to remove unhealthy nodes”
4. “Scale LB horizontally for HA”

---

### Q: “How to handle failures?”

**Answer:**

* Multiple LBs
* Health checks
* Failover via DNS/Anycast

---

### Q: “How to handle user sessions?”

**Answer:**

* Prefer stateless design
* Store session in Redis
* Avoid sticky sessions if possible
