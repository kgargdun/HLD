## **Load Balancing Fundamentals & Mental Model.md**

---

## **1.1 What is Load Balancing (Interview Framing)**

**Definition (say this clearly):**
Load balancing is the process of distributing incoming traffic across multiple servers to improve availability, reduce latency, and increase throughput.

---

### Core Goals

* **Scalability** → handle more traffic
* **Availability** → avoid single point of failure
* **Performance** → reduce response time

---

### Simple Mental Model

```
Client → Load Balancer → Multiple Servers
```

Instead of:

```
Client → Single Server (bottleneck ❌)
```

---

**Interview Line:**

> “Load balancer acts as a traffic distributor to prevent overload and improve system reliability.”

---

## **1.2 Why Do We Need Load Balancing**

### 1. Handle High Traffic

* Single server cannot scale infinitely

---

### 2. Fault Tolerance

* If one server fails → traffic routed to others

---

### 3. Better Resource Utilization

* Avoid some servers being overloaded while others idle

---

### 4. Zero Downtime Deployments

* Gradually shift traffic to new version

---

**Real Example:**

* Instagram feed service handling millions of requests/sec

---

## **1.3 Horizontal vs Vertical Scaling**

### Vertical Scaling (Scale Up)

* Increase CPU/RAM of single machine

**Limits:**

* Hardware ceiling
* Expensive

---

### Horizontal Scaling (Scale Out)

* Add more machines

**Requires:**

* Load balancing

---

**Interview Insight:**

> “Load balancing is essential for horizontal scaling.”

---

## **1.4 Types of Traffic**

### HTTP / HTTPS

* Most common (web apps)

---

### TCP

* Database connections
* Persistent connections

---

### UDP

* Gaming, streaming

---

**Why it matters:**

* Determines whether you use **L4 or L7 LB**

---

## **1.5 Where Load Balancer Sits**

### Typical Architecture

```
Client
  ↓
CDN (optional)
  ↓
Load Balancer
  ↓
Application Servers
  ↓
Database
```

---

### Multiple Layers of LB

* Edge LB (public)
* Internal LB (service-to-service)

---

**Interview Line:**

> “We can have multiple layers of load balancing for better scalability and isolation.”

---

## **1.6 Key Metrics (Interview Gold)**

### Latency

* Time taken to serve request

---

### Throughput

* Requests per second

---

### Error Rate

* Failed requests

---

### Server Utilization

* CPU, memory usage

---

### Tail Latency (IMPORTANT)

* p95 / p99 latency

---

**Interview Insight:**

> “Optimizing average latency is not enough — tail latency matters in distributed systems.”

---

## **1.7 Types of Load Balancers (High-Level)**

### Hardware Load Balancers

* Dedicated devices

**Cons:** expensive, less flexible

---

### Software Load Balancers

* Run on servers

**Examples:**

* NGINX
* HAProxy

---

### Cloud Load Balancers

* Managed services

**Example:**

* AWS ELB

---

## **1.8 When NOT to Use Load Balancer**

* Very small systems
* Single-user tools
* Low traffic services

---

## **Mental Model (Most Important)**

Think of load balancer as:

> “A smart router that distributes traffic based on system state and strategy.”

---

Always ask:

1. How is traffic distributed?
2. What happens if a server fails?
3. How does LB know server health?
4. Is routing stateless or stateful?

---

## **How to Use in Interviews**

When designing system:

1. Start with single server
2. Say:

   > “To scale, I’ll introduce a load balancer in front of application servers”
3. Then expect:

   * “Which algorithm?”
   * “How handle failures?”
   * “L4 or L7?”

---

## **Common Mistakes**

* Not mentioning load balancer at all
* Ignoring failure handling
* Not distinguishing L4 vs L7
* Treating LB as “just routing” (it’s smarter)

---

## **Golden Rule**

> “Load balancer enables scaling, but poor strategy can still overload your system.”
