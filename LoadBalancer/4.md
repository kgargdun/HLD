## **Distributed Load Balancing, Scaling & Failures.md**

---

## **4.1 Single Load Balancer = Bottleneck**

### Problem

```
Client → Load Balancer → Servers
```

* LB becomes:

  * Single point of failure ❌
  * Throughput bottleneck ❌

---

### Fix

```
        → LB1 →
Client → LB2 → → Servers
        → LB3 →
```

---

**Interview Line:**

> “A single load balancer is itself a SPOF, so we need multiple LBs.”

---

## **4.2 Multiple Load Balancers (HA Setup)**

### Active–Passive

* One LB active
* One standby

**Pros**

* Simple

**Cons**

* Failover delay

---

### Active–Active

* All LBs serve traffic

**Pros**

* Better utilization
* No idle resources

**Cons**

* More complex

---

## **4.3 How Do Clients Reach Multiple LBs?**

### 1. DNS-Based Load Balancing

* DNS returns multiple IPs

```
example.com → [IP1, IP2, IP3]
```

---

### Pros

* Simple
* No extra infrastructure

---

### Cons (IMPORTANT)

* DNS caching → slow failover
* No real-time health awareness

---

### 2. Anycast Routing

* Same IP advertised from multiple locations
* Network routes to nearest LB

---

### Pros

* Low latency
* Automatic failover

---

### Cons

* Complex setup

---

**Interview Line:**

> “DNS is simple but slow to react, Anycast gives better latency and failover.”

---

## **4.4 Health Checks (Critical)**

### Why needed?

* LB must know which servers are healthy

---

### Types

#### Active Health Checks

* LB periodically pings servers

---

#### Passive Health Checks

* Detect failures from live traffic

---

### What to check?

* HTTP status (200 OK)
* Response time
* Custom health endpoints

---

**Interview Insight:**

> “Without health checks, load balancer may route traffic to dead servers.”

---

## **4.5 Failover Strategies**

### Server Failure

* LB removes unhealthy server from pool

---

### Load Balancer Failure

* Traffic routed to other LBs (via DNS/Anycast)

---

### Zone Failure (Advanced)

* Multi-AZ deployment

---

**Interview Line:**

> “System should survive instance, LB, and even AZ failures.”

---

## **4.6 Sticky Sessions (Session Affinity)**

### Problem

* User session stored on specific server

---

### Solution

* Route same user → same server

---

### Methods

#### 1. IP Hash

* Same IP → same server

---

#### 2. Cookies (L7)

* LB injects session cookie

---

### Tradeoff

| Benefit             | Problem     |
| ------------------- | ----------- |
| Session consistency | Uneven load |

---

**Better Approach:**

* Store session in shared cache (e.g., Redis)

---

## **4.7 Circuit Breaking (Intro)**

### Problem

* Failing service keeps receiving traffic

---

### Solution

* Temporarily stop sending requests

---

### States

* Closed → normal
* Open → blocked
* Half-open → retry

---

**Interview Insight:**

> “Prevents cascading failures in distributed systems.”

---

## **4.8 Rate Limiting + Load Balancer**

### Why combine?

* Protect backend from overload

---

### Example

* Limit requests per IP

---

### Implementation

* At LB layer (efficient)
* Using Redis counters

---

## **4.9 Scaling Load Balancers**

### Vertical Scaling

* Bigger LB machine

---

### Horizontal Scaling

* Add more LB nodes

---

### Auto Scaling

* Add/remove LBs based on traffic

---

**Cloud Example:**

* AWS ELB auto-scales

---

## **4.10 Common Failure Scenarios**

### 1. LB Overload

* Too much traffic → LB itself slows

---

### 2. Uneven Distribution

* Bad algorithm → hotspots

---

### 3. Health Check Misconfiguration

* Healthy servers marked unhealthy (or vice versa)

---

### 4. Cascading Failures

* One service fails → overload others

---

## **Tradeoffs**

| Aspect     | Option A  | Option B     |
| ---------- | --------- | ------------ |
| Simplicity | Single LB | Multiple LBs |
| Failover   | DNS       | Anycast      |
| Sessions   | Sticky    | Stateless    |
| Scaling    | Manual    | Auto         |

---

## **Interview Playbook**

When asked:

> “How do you make load balancer highly available?”

Answer:

1. “Use multiple LBs (active-active)”
2. “Route via DNS or Anycast”
3. “Add health checks”
4. “Enable failover”
5. “Keep system stateless if possible”

---

## **Golden Rule**

> “Load balancer should never become the bottleneck or single point of failure.”
