# Communication Patterns (Decision Layer)

---

## 🧠 Core Mental Model
> Same problem can be solved using multiple communication styles

👉 Choosing the right one = key system design skill

---

# 🔄 Request-Response vs Streaming vs Real-Time

| Pattern | Description | Use Case |
|--------|------------|----------|
| Request-Response | One request → one response | APIs |
| Streaming | Continuous data flow | logs, analytics |
| Real-time | Instant bidirectional | chat, gaming |

---

# 🌐 REST vs gRPC vs GraphQL

---

## 🎯 REST

### What
- Resource-based APIs over HTTP
- Uses JSON

---

### How it works


GET /users/1
POST /orders



---

### Pros
- Simple  
- Widely adopted  

---

### Cons
- Over-fetching / under-fetching  
- Multiple requests needed  

---

---

## ⚡ gRPC

---

## 🎯 What
- High-performance RPC framework

---

## ⚙️ How it works
- Uses **Protocol Buffers (binary)**
- Runs over HTTP/2
- Strongly typed contracts

---

### Pros
- Fast (binary format)  
- Supports streaming  
- Contract-first  

---

### Cons
- Harder debugging  
- Less browser support  

---

---

## 🔍 GraphQL

---

## 🎯 What
- Client specifies exactly what data it needs

---

## ⚙️ How it works


query {
user {
name
posts {
title
}
}
}


---

### Pros
- No over-fetching  
- Flexible  

---

### Cons
- Complex backend  
- Hard caching  

---

---

## ⚖️ REST vs gRPC vs GraphQL

| Feature | REST | gRPC | GraphQL |
|--------|------|------|--------|
| Format | JSON | Binary | JSON |
| Performance | Medium | High | Medium |
| Flexibility | Low | Low | High |
| Streaming | ❌ | ✅ | Limited |
| Best for | Public APIs | Internal services | UI-driven apps |

---

### 🔥 Memory Hook
> REST = simple  
> gRPC = fast  
> GraphQL = flexible  

---

# 🔁 Polling vs SSE vs WebSocket

---

## 📡 Polling

### What
- Client repeatedly asks server

---

### Problem
- Wasteful  
- High latency  

---

---

## ⏳ Long Polling

### What
- Server holds request until data available

---

### Improvement over polling
- Reduces unnecessary calls  

---

---

## 📢 Server-Sent Events (SSE)

---

### What
- Server → client streaming (one-way)

---

### Pros
- Simpler than WebSocket  
- Auto-reconnect  

---

### Cons
- One-way only  

---

---

## 🔌 WebSocket

---

### What
- Full duplex persistent connection

---

### Pros
- Real-time  
- Low latency  

---

### Cons
- Stateful  
- Scaling complexity  

---

---

## ⚖️ Comparison

| Feature | Polling | SSE | WebSocket |
|--------|--------|-----|----------|
| Direction | Request-based | Server → Client | Bidirectional |
| Latency | High | Medium | Low |
| Complexity | Low | Medium | High |
| Use case | Simple apps | Notifications | Chat, gaming |

---

### 🔥 Memory Hook
> Polling = waste  
> SSE = push  
> WebSocket = real-time  

---

# ⚖️ L4 vs L7 Load Balancing

---

## 📡 L4 (Transport Layer)

- Based on IP + port  
- Fast  
- No content awareness  

---

## 🌐 L7 (Application Layer)

- Based on URL, headers  
- Smart routing  
- Supports:
  - auth
  - caching  

---

---

## ⚖️ Comparison

| Feature | L4 | L7 |
|--------|----|----|
| Speed | Fast | Slower |
| Intelligence | Low | High |
| Routing | Basic | Advanced |

---

### 🔥 Memory Hook
> L4 = fast  
> L7 = smart  

---

# 🔁 Sync vs Async Communication

---

## 🔄 Synchronous

- Client waits for response  

### Example
- REST API call  

---

## ⏳ Asynchronous

- Client doesn’t wait  

### Example
- background processing  

---

### ⚠️ Note
(Message queues covered separately)

---

## ⚖️ Comparison

| Feature | Sync | Async |
|--------|------|------|
| Latency | Higher | Lower |
| Coupling | Tight | Loose |
| Complexity | Low | High |

---

# 🧠 Real-World Decision Mapping

| Problem | Solution |
|--------|---------|
| Public API | REST |
| Internal microservices | gRPC |
| UI with dynamic data | GraphQL |
| Notifications | SSE |
| Chat system | WebSocket |
| High throughput routing | L4 LB |
| Smart routing | L7 LB |

---

# 🔥 Interview Insights

- REST is default, not always optimal  
- gRPC reduces latency in microservices  
- WebSockets introduce state → harder scaling  
- GraphQL shifts complexity to backend  
- L7 load balancers enable intelligent routing  

---

# 🧠 Quick Recap

- Multiple ways to communicate  
- Choose based on:
  - latency
  - flexibility
  - complexity  

---

# 🔥 One-Liners

- “Choice of protocol depends on latency and flexibility requirements”
- “gRPC is preferred for internal low-latency communication”
- “WebSockets are useful for real-time bidirectional systems”
- “GraphQL reduces over-fetching at the cost of backend complexity”
