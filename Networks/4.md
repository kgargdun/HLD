```md id="net-app-protocols-04"
# Core Application Protocols

---

## 🧠 Core Mental Model
> Application layer defines HOW applications communicate

- Built on top of TCP/UDP
- Defines:
  - data format
  - request/response model
  - communication pattern

---

# 🌐 HTTP (HyperText Transfer Protocol)

---

## 🎯 What problem?
- Standard way for client ↔ server communication

---

## ⚙️ How it works

1. Client sends request
2. Server processes
3. Server sends response

```

Client → Request → Server → Response

```id="q6n1pz"

---

## 📦 Structure

### Request
- Method (GET, POST, etc.)
- Headers
- Body

### Response
- Status code
- Headers
- Body

---

## ⚠️ Key Properties

- Stateless  
- Text-based (HTTP/1.1)  
- Runs over TCP  

---

## 🔁 HTTP Versions

### HTTP/1.1
- One request per connection  
- Head-of-line blocking  

---

### HTTP/2
- Multiplexing (multiple requests over one connection)  
- Header compression  

---

### HTTP/3
- Runs over UDP (QUIC)  
- Avoids TCP head-of-line blocking  

---

### 🔥 Memory Hook
> HTTP evolution = reduce latency + remove blocking  

---

## ⚖️ Trade-offs

- ✅ Simple  
- ✅ Widely supported  
- ❌ Overhead (headers, text format)  

---

## 📍 Where used
- REST APIs  
- Web applications  

---

# 🔌 WebSocket

---

## 🎯 What problem?
- Real-time bidirectional communication

---

## ⚙️ How it works

1. Starts as HTTP request  
2. Upgrades connection  
3. Persistent full-duplex channel  

```

HTTP → Upgrade → WebSocket (persistent connection)

```id="m2k9pl"

---

## ⚠️ Key Properties

- Full duplex  
- Persistent connection  
- Low latency  

---

## ⚖️ Trade-offs

- ✅ Real-time  
- ❌ Stateful (harder to scale)  

---

## 📍 Where used
- Chat apps  
- Live updates  
- Gaming  

---

### 🔥 Memory Hook
> WebSocket = always-open pipe  

---

# 🔐 TLS / HTTPS

---

## 🎯 What problem?
- Secure communication over network

---

## ⚙️ How it works (simplified)

1. Client connects  
2. Server sends certificate  
3. Key exchange (asymmetric)  
4. Secure channel (symmetric encryption)  

---

## ⚠️ Key Properties

- Encryption  
- Authentication  
- Integrity  

---

### 🔥 Memory Hook
> TLS = identity + encryption  

---

## 📍 Where used
- HTTPS  
- Secure APIs  

---

# 🌍 DNS (Intro)

---

## 🎯 What problem?
- Human-readable names → IP addresses

---

## ⚙️ Basic Flow

1. Client queries DNS  
2. Resolver finds IP  
3. Returns result  

---

👉 Deep dive in DNS file

# ⚠️ Key Comparisons

---

## HTTP vs WebSocket

| Feature | HTTP | WebSocket |
|--------|------|----------|
| Type | Request-response | Full duplex |
| Connection | Short-lived | Persistent |
| Use case | APIs | Real-time |

---

## HTTP vs HTTPS

| Feature | HTTP | HTTPS |
|--------|------|--------|
| Security | ❌ | ✅ |
| Encryption | ❌ | ✅ |
| Port | 80 | 443 |

---

## 🧠 Real-World Mapping

| Protocol | Usage |
|----------|------|
| HTTP | REST APIs |
| WebSocket | Chat systems |
| TLS | Secure communication |
| DNS | Every request |

---

## 🔥 Interview Insights

- HTTP is stateless → need external state (DB/cache)
- WebSocket introduces state → scaling complexity
- TLS adds latency (handshake) but required for security
- HTTP/2 reduces connection overhead
- HTTP/3 improves latency using UDP

---

## 🧠 Quick Recap

- HTTP → request-response  
- WebSocket → real-time communication  
- TLS → secure layer  
- DNS → name resolution  

---

## 🔥 One-Liners

- “HTTP is stateless, so we externalize state”
- “WebSockets enable low-latency real-time communication”
- “TLS ensures secure communication over untrusted networks”
- “HTTP/2 multiplexing improves performance”
- “HTTP/3 removes TCP-level bottlenecks”
```
