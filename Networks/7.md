# Specialized / Niche Protocols

---

## 🧠 Core Mental Model
> Not all systems use HTTP

- Some systems need:
  - low bandwidth
  - async communication
  - device-level reliability

👉 These protocols solve **specific constraints**

---

# 📡 MQTT (Message Queuing Telemetry Transport)

---

## 🎯 What problem?
- Communication over:
  - low bandwidth
  - unreliable networks
  - constrained devices (IoT)

---

## ⚙️ How it works

- **Publish-Subscribe model**
- Central **broker**



Publisher → Broker → Subscribers

id="mqtt-flow-1"

---

### Flow

1. Client connects to broker  
2. Publisher sends message to topic  
3. Broker forwards to subscribers  

---

## 📦 Key Concepts (VERY IMPORTANT)

---

### 1. Topics
- Logical channels
- Example:
  - home/temperature
  - device/123/status

---

### 2. QoS (Quality of Service)

| Level | Guarantee |
|------|----------|
| 0 | At most once |
| 1 | At least once |
| 2 | Exactly once |

---

### 3. Retained Messages
- Broker stores last message  
- New subscribers receive it immediately  

---

### 4. Last Will (LWT)
- Sent if client disconnects unexpectedly  
- Used for failure detection  

---

## ⚖️ Trade-offs

- ✅ Lightweight  
- ✅ Efficient for IoT  
- ✅ Handles unreliable networks  
- ❌ Requires broker  
- ❌ Not ideal for general APIs  

---

## 📍 Where used
- IoT systems  
- telemetry pipelines  
- device communication  

---

### 🔥 Memory Hook
> MQTT = lightweight pub-sub for unreliable networks  

---

---

# 📡 AMQP (Advanced Message Queuing Protocol)

---

## 🎯 What problem?
- Reliable message delivery between systems

---

## ⚙️ How it works

- Broker-based  
- Uses:
  - exchanges
  - queues
  - routing keys  

---

### Flow



Producer → Exchange → Queue → Consumer

id="amqp-flow-1"

---

## ⚠️ Key Properties

- Reliable delivery  
- Supports acknowledgements  
- Flexible routing  

---

## ⚖️ Trade-offs

- ✅ Reliable  
- ✅ Flexible routing  
- ❌ Heavier than MQTT  

---

## 📍 Where used
- Backend systems  
- message queues (e.g., RabbitMQ)  

---

### 🔥 Memory Hook
> AMQP = reliable enterprise messaging  

---

---

# 📡 XMPP (Extensible Messaging and Presence Protocol)

---

## 🎯 What problem?
- Real-time messaging + presence (online/offline)

---

## ⚙️ How it works

- XML-based protocol  
- Uses **stanzas**:
  - message
  - presence
  - iq  

---

### Flow

- Client → server  
- Server routes to recipient  
- Supports federation (server ↔ server)

---

## ⚠️ Key Properties

- Real-time messaging  
- Presence tracking  
- Federated architecture  

---

## ⚖️ Trade-offs

- ✅ Standardized  
- ❌ XML overhead  
- ❌ Less efficient than modern protocols  

---

## 📍 Where used
- legacy chat systems  
- some enterprise messaging  

---

### 🔥 Memory Hook
> XMPP = XML-based chat protocol (legacy-ish)  

---

---

# ⚖️ Comparison Summary

| Feature | MQTT | AMQP | XMPP |
|--------|------|------|------|
| Model | Pub-sub | Queue-based | Messaging |
| Weight | Light | Heavy | Medium |
| Reliability | QoS-based | Strong | Medium |
| Best for | IoT | Backend systems | Chat |

---

---

# 🧠 When to Use (Interview Mapping)

| Problem | Protocol |
|--------|---------|
| IoT devices | MQTT |
| Reliable backend messaging | AMQP |
| Chat (legacy systems) | XMPP |

- MQTT is ideal for constrained environments  
- AMQP ensures strong delivery guarantees  
- XMPP is mostly replaced by modern protocols
