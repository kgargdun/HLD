# README 4 — Scaling the System & Real-Time Updates

## 6. Distributed Geo Systems (Scaling Layer)

### 6.1 Why Distribution is Needed

Single machine breaks when:

* millions of users
* high QPS (reads + writes)
* real-time updates

Goal:

> Distribute data + queries across multiple nodes efficiently

---

### 6.2 Partitioning Strategies

#### a) Geo-Based Partitioning (Most Intuitive)

* Divide world into regions
* Each region handled by a server

Example:

* India → shard A
* US → shard B

---

#### Advantages:

* locality preserved
* queries usually hit nearby shard

---

#### Problems:

* uneven load (cities vs rural)
* hotspots (dense regions overload single shard)

---

### 6.3 Hash-Based Partitioning

* Hash geohash / location → shard

---

#### Advantages:

* uniform load distribution
* avoids hotspots

---

#### Problems:

* destroys locality
* queries must hit multiple shards

---

### 6.4 Hybrid Approach (Real Systems)

Combine both:

* coarse geo partition (region-level)
* within region → hash or smaller cells

---

### 6.5 Sharding with Geohash Prefix

Idea:

* use geohash prefix as shard key

Example:

* prefix `tdr5` → shard 1
* prefix `tdr6` → shard 2

---

#### Benefit:

* preserves locality
* easy routing

---

### 6.6 Hotspot Problem (Critical)

Dense areas:

* thousands of users in same cell

Result:

* single shard overloaded

---

#### Solutions:

* finer granularity (longer geohash)
* dynamic splitting of hot regions
* load-aware routing

---

### 6.7 Cross-Shard Queries

Problem:

* neighbors may lie in different shards

---

#### Handling:

* query multiple shards in parallel
* merge results

---

### 6.8 Replication Strategy

* replicate nearby regions across nodes
* improves read latency
* provides fault tolerance

---

### 6.9 Practical Insight

Real systems prioritize:

* **locality for reads**
* **balanced load for writes**

Tradeoff is unavoidable → hybrid design dominates

---

## 7. Real-Time Location Ingestion (Write Path)

### 7.1 Nature of the Problem

* continuous location updates
* high write throughput
* moving entities

---

### 7.2 Data Flow (High-Level)

1. Client sends GPS updates
2. Backend ingests updates
3. Location stored / indexed
4. Index updated

---

### 7.3 GPS Update Characteristics

* noisy (inaccurate)
* frequent (every few seconds)
* bursty (network conditions)

---

### 7.4 Update Frequency Control

Problem:

* too many updates overwhelm system

---

#### Techniques:

* threshold-based updates (distance moved)
* time-based throttling
* adaptive frequency

---

### 7.5 Write-Heavy System Challenges

* high throughput
* frequent index updates
* consistency issues

---

### 7.6 Streaming Systems (Backbone)

Used for ingesting location streams:

* Apache Kafka
* Apache Flink

---

### 7.7 Role of Streaming

#### Kafka:

* durable ingestion buffer
* decouples producers & consumers

#### Flink:

* real-time processing
* can update indexes / trigger actions

---

### 7.8 Handling Moving Objects

Key issue:

> Location constantly changes → index becomes stale

---

#### Approaches:

##### a) Update-in-place

* overwrite old location

##### b) Append-only (event log)

* store history
* compute latest dynamically

---

### 7.9 Consistency vs Freshness

* strong consistency → slower
* eventual consistency → slight staleness

Most systems choose:

> slightly stale but fast

---

### 7.10 Practical System Insight

Typical pipeline:

Client → API → Kafka → Stream Processor → Geo Index (Redis / DB)

* writes are async
* reads hit fast index

---

### 7.11 Hidden Insight (Critical)

Write path and read path are **decoupled**

> Writes optimize throughput
> Reads optimize latency

They do NOT use same system design.
