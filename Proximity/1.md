# README 1 — Foundations of Proximity Detection

## 1. Problem Framing & Query Types

### 1.1 What is Proximity Detection?

At its core:

> Given a location (latitude, longitude), find relevant nearby entities.

Entities can be:

* users (friends, riders, players)
* service providers (drivers, delivery agents)
* physical locations (stores, restaurants, ATMs)

---

### 1.2 Core Query Types

#### a) Nearest Neighbor (kNN)

* Find **k closest entities** to a point
* Example:

  * Find 5 nearest drivers to a rider

---

#### b) Range Query (Radius Search)

* Find all entities within distance **R**
* Example:

  * Restaurants within 3 km

---

#### c) Hybrid Queries

* First filter by radius, then rank
* Example:

  * Restaurants within 5 km sorted by rating

---

### 1.3 Relationship Types

#### One-to-One

* Single closest match
* Example:

  * Assign nearest driver

#### One-to-Many

* One user → multiple results
* Example:

  * Nearby stores

#### Many-to-Many

* Matching between two moving sets
* Example:

  * Drivers ↔ Riders

---

### 1.4 Static vs Dynamic Entities

#### Static

* Location rarely changes
* Examples:

  * stores, warehouses

#### Dynamic (Hard Case)

* Continuously moving
* Examples:

  * drivers, users

Key challenge:

* Index becomes stale quickly

---

### 1.5 Real-Time vs Offline

#### Real-Time

* Query must reflect current position
* Latency critical (10–100 ms)

#### Offline / Batch

* Precompute results
* Used for analytics or recommendations

---

### 1.6 Accuracy Requirements

#### Exact

* True nearest neighbors
* High compute cost

#### Approximate

* “Close enough”
* Much faster
* Often acceptable in real systems

---

### 1.7 Latency Constraints

* Ride-hailing → ultra low latency
* Search (restaurants) → moderate latency
* Analytics → high latency acceptable

---

### 1.8 Update Frequency

* High frequency (every few seconds)

  * drivers, delivery agents
* Medium frequency

  * mobile users
* Low frequency

  * stores

Implication:

* Write-heavy vs read-heavy system design

---

## 2. Geo Fundamentals & Distance Computation

### 2.1 Latitude & Longitude

* Latitude: North-South position (−90 to +90)
* Longitude: East-West position (−180 to +180)

Represents a point on Earth’s surface.

---

### 2.2 Earth Model

#### Sphere (Simplified)

* Easier math
* Slight inaccuracy

#### Ellipsoid (Real Earth)

* More accurate
* More complex calculations

---

### 2.3 Distance Computation Methods

#### a) Euclidean Distance (Flat Approximation)

Used when:

* small distances
* high performance required

Formula:

* Treat lat/lon as x,y

Problem:

* inaccurate over large distances

---

#### b) Haversine formula

Used for:

* accurate distance on sphere

Key idea:

* shortest path over Earth’s surface (great-circle distance)

Properties:

* more accurate than Euclidean
* slightly expensive computationally

---

#### c) Vincenty Formula

Used for:

* highest accuracy (ellipsoidal Earth)

Tradeoff:

* more computation
* rarely needed in most applications

---

### 2.4 Precision vs Performance Tradeoff

| Method    | Accuracy | Speed  | Use Case                     |
| --------- | -------- | ------ | ---------------------------- |
| Euclidean | Low      | Fast   | small areas, rough filtering |
| Haversine | Medium   | Medium | most real-world systems      |
| Vincenty  | High     | Slow   | high-precision systems       |

---

### 2.5 Practical System Insight

Real systems almost always:

1. Use **cheap approximation first** (filter candidates)
2. Then apply **accurate distance** on reduced set

This two-step approach is critical for scalability.

---

### 2.6 Hidden Insight (Important)

Distance calculation is **not the bottleneck at scale**.

The real problem is:

> “How do we avoid computing distance for millions of points?”

This leads to:

* spatial indexing
* partitioning
* pruning search space

(covered in next README)
