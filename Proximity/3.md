# README 3 — Encoding Space & Query Execution

## 5. Geohashing & Space-Filling Curves

### 5.1 Why We Need Encoding

So far:

* We partitioned space (grids, trees)

Problem:

* How do we **represent location as a key** that works well in databases and distributed systems?

Goal:

> Convert (lat, lon) → compact, searchable identifier

---

### 5.2 Geohash (Most Practical Approach)

#### Idea:

* Encode latitude & longitude into a **single string**
* Nearby locations share **common prefixes**

Example (conceptual):

* Location A → `tdr5r`
* Nearby → `tdr5x`
* Same prefix (`tdr5`) ⇒ close proximity

---

### 5.3 How Geohash Works (Internal)

1. Latitude and longitude ranges:

   * Lat: [-90, +90]
   * Lon: [-180, +180]

2. Recursively divide ranges:

   * Split into halves
   * Track whether point is in lower/upper half

3. Interleave bits:

   * lon bit, lat bit, lon bit, lat bit...

4. Convert binary → Base32 string

---

### 5.4 Precision Levels

* Longer geohash → smaller area

| Length | Area Size |
| ------ | --------- |
| 5      | ~5 km     |
| 6      | ~1 km     |
| 7      | ~150 m    |
| 8      | ~40 m     |

---

### 5.5 Key Property (Very Important)

> Prefix similarity ≈ spatial proximity

This enables:

* prefix search
* range queries using strings

---

### 5.6 Neighbor Problem

Issue:

* Adjacent locations may fall into **different geohashes**

Example:

* Border of a cell

Solution:

* Query **neighboring geohashes** as well

(typically 8 neighbors in grid)

---

### 5.7 False Positives

Geohash gives:

* approximate region

Problem:

* Some returned points may not actually be closest

Solution:

* post-filter using exact distance

---

### 5.8 Space-Filling Curves

Alternative to geohash:

#### a) Z-Order Curve (Morton Code)

* Interleave bits of lat/lon
* Maps 2D → 1D

#### b) Hilbert Curve

* Better locality preservation
* More complex

---

### 5.9 Why These Matter

They allow:

* storing geo data in sorted structures
* range queries using 1D indexes

---

### 5.10 Practical Insight

Real systems prefer:

* geohash-like approaches

Because:

* simple
* works with key-value stores
* easy sharding

---

## 8. Query Execution Pipeline (Read Path)

### 8.1 Core Goal

> Efficiently find nearest entities without scanning everything

---

### 8.2 Step-by-Step Pipeline

#### Step 1: Encode Query Location

* Convert (lat, lon) → geohash

---

#### Step 2: Fetch Candidate Cells

* Start with same cell
* Add neighboring cells

---

#### Step 3: Candidate Retrieval

* Fetch all entities in those cells

---

#### Step 4: Distance Filtering

* Compute accurate distance (e.g., Haversine)
* Remove far-away points

---

#### Step 5: Ranking

* Sort by distance
* Return top k

---

### 8.3 Expanding Radius Strategy

Problem:

* What if not enough results?

Solution:

* Expand search area:

  * include more neighboring cells
  * repeat until enough candidates

---

### 8.4 Candidate Generation vs Filtering

Two phases:

#### Candidate Generation (cheap)

* geohash / grid lookup
* fast, approximate

#### Candidate Filtering (expensive)

* precise distance computation

---

### 8.5 kNN Search Strategy

1. Start small region
2. Check results count
3. Expand region progressively
4. Stop when k results found with confidence

---

### 8.6 Radius Query Strategy

* Directly fetch all cells intersecting radius
* Then filter precisely

---

### 8.7 Approximate vs Exact

#### Approximate

* Faster
* May miss exact nearest

#### Exact

* Expand until correctness guaranteed
* Higher latency

---

### 8.8 Latency Optimization Techniques

* Limit max expansion
* Early stopping
* Pre-sorted cell data
* Cache hot regions

---

### 8.9 Hidden Insight (Critical)

The real system is NOT:

> “Find nearest point”

It is:

> “Find small candidate set cheaply → refine precisely”

Everything revolves around:

* minimizing candidate size
* minimizing expansions
