# README 2 — From Naive to Efficient Proximity Search

## 3. Baseline Approaches (Naive → Limits)

### 3.1 Linear Scan (Brute Force)

**Idea:**

* For a query point (lat, lon):

  * compute distance to every entity
  * keep the closest (or top k)

**Steps:**

1. Iterate over all points (n)
2. Compute distance (e.g., Haversine)
3. Maintain best result (min / heap for kNN)

**Time Complexity:**

* O(n) per query

---

### 3.2 Sorting-Based Approach

**Idea:**

* Compute all distances
* Sort results

**Time Complexity:**

* O(n log n)

**Observation:**

* Worse than linear scan for kNN
* Only useful if full ordering is needed

---

### 3.3 Why Naive Works (Sometimes)

Naive is acceptable when:

* n is small (few thousand)
* queries are rare
* data is static
* offline processing

---

### 3.4 Why Naive Fails at Scale

#### a) Latency Explosion

* 1M points → 1M distance calculations per query

#### b) CPU Bottleneck

* expensive trig operations (Haversine)

#### c) Cache Inefficiency

* memory scans are not cache-friendly

#### d) Poor Scalability

* QPS × n grows uncontrollably

---

### 3.5 Key Insight (Critical)

> Problem is NOT computing distance
> Problem is reducing the candidate set

This is the transition point to spatial indexing.

---

## 4. Spatial Indexing (Core Data Structures)

### 4.1 Goal of Spatial Indexing

> Avoid scanning all points
> Only examine “relevant regions”

---

### 4.2 Grid-Based Indexing (Simplest & Most Important)

#### Idea:

* Divide space into fixed-size cells
* Map each point → a cell

#### Structure:

* Key: (cell_x, cell_y)
* Value: list of points in that cell

---

#### Query Flow:

1. Find cell of query point
2. Check that cell
3. Expand to neighboring cells if needed

---

#### Advantages:

* Extremely fast lookup
* Simple to implement
* Works well for real-time systems

---

#### Problems:

* Choosing grid size is hard
* Uneven distribution → hotspots
* Boundary issues (edge of cells)

---

### 4.3 Hierarchical Grids

#### Idea:

* Multiple grid levels (coarse → fine)

#### Benefit:

* Adaptive resolution
* Better control over density

---

### 4.4 k-d Tree

#### Idea:

* Binary tree splitting space alternately (x, y)

#### Properties:

* Balanced partitioning
* Efficient nearest neighbor search

---

#### Query:

* Traverse tree
* Prune subtrees that cannot contain closer points

---

#### Limitations:

* Poor for dynamic updates
* Rebalancing expensive

---

### 4.5 Quad Tree

#### Idea:

* Recursively divide space into 4 quadrants

---

#### Properties:

* Adaptive partitioning
* Dense regions → deeper trees

---

#### Good for:

* Non-uniform distributions

---

#### Issues:

* Pointer-heavy (memory overhead)
* Not great for distributed systems directly

---

### 4.6 R-Tree / R*-Tree

#### Idea:

* Group nearby objects into bounding rectangles

---

#### Structure:

* Each node = bounding box
* Leaf nodes = actual objects

---

#### Query:

* Traverse nodes whose bounding boxes intersect query region

---

#### Advantages:

* Industry standard for geo databases
* Efficient range queries

---

#### Used in:

* PostGIS
* MySQL spatial indexes

---

### 4.7 Core Concept: Pruning

All spatial indexes rely on:

> “Eliminate large regions that cannot contain answers”

Examples:

* Skip entire grid cells
* Skip subtrees in k-d tree
* Skip bounding boxes in R-tree

---

### 4.8 Comparison (Intuition-Level)

| Structure | Best For               | Weakness        |
| --------- | ---------------------- | --------------- |
| Grid      | Real-time, simple      | tuning size     |
| k-d Tree  | Static datasets        | dynamic updates |
| Quad Tree | Uneven distributions   | memory overhead |
| R-Tree    | Databases, range query | complex         |

---

### 4.9 Practical System Insight

Real-world systems often:

* Prefer **grid / geohash style indexing**
* Avoid heavy tree structures in distributed systems

Reason:

* easier sharding
* simpler scaling
* predictable performance

---

### 4.10 Hidden Insight (Important)

Spatial indexing is essentially:

> Converting a 2D search problem → smaller localized searches

The entire system design will revolve around:

* how we partition space
* how we search nearby partitions
