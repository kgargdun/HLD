# Data Modeling Fundamentals

## 1. Core Concepts

### Entities, Attributes, Relationships

* **Entity**: Real-world object (User, Order, Movie)
* **Attributes**: Properties of an entity (user_id, name, email)
* **Relationship**: Association between entities (User → Orders)

### Cardinality

* **1:1** → One user has one profile
* **1:N** → One user has many orders
* **N:M** → Many users watch many movies (requires join table)

### Keys

* **Primary Key (PK)**: Unique identifier (user_id)
* **Candidate Key**: Possible unique identifiers
* **Composite Key**: Multiple columns form uniqueness (user_id + movie_id)

---

## 2. Normalization

### 1NF (First Normal Form)

* No repeating groups
* Atomic values only

### 2NF (Second Normal Form)

* No partial dependency on composite key

### 3NF (Third Normal Form)

* No transitive dependency (non-key → non-key)

### Why Normalize?

* Eliminates redundancy
* Improves consistency

### When NOT to Normalize

* Read-heavy systems
* High-performance requirements
* Leads to excessive joins

---

## 3. Design Thinking (Most Important)

### Access Pattern First

* Always ask:

  * What queries will run?
  * What are the hot paths?

Example:

* Fetch user orders → design for fast lookup by user_id

---

### Read vs Write Optimization

| Aspect   | Read Heavy   | Write Heavy   |
| -------- | ------------ | ------------- |
| Design   | Denormalized | Normalized    |
| Goal     | Fast reads   | Fast writes   |
| Tradeoff | Redundancy   | Complex reads |

---

### Query-Driven Design

* Schema should support:

  * Frequent queries efficiently
  * Minimal joins (at scale)
* Avoid designing purely from ER diagrams

---

## 4. Key Tradeoffs

### Normalization vs Denormalization

* Normalized → clean, consistent
* Denormalized → fast, scalable

### Flexibility vs Performance

* Flexible schema → slower queries
* Strict schema → optimized queries

---

## 5. Common Interview Mistakes

* Designing schema without knowing queries
* Over-normalizing high-scale systems
* Ignoring read/write patterns
* Not thinking about future scale

---

## TL;DR

* Start with **access patterns**
* Use **normalization for correctness**
* Use **denormalization for performance**
* Always design for **scale + queries**, not just structure
