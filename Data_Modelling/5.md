# Data + API Integration (End-to-End)

## 1. End-to-End Flow

### Basic Flow

* Client → API → Service → Database → Response

### Expanded Flow (Real Systems)

* Client → API Gateway → Service → Cache/DB → Response

---

## 2. Layered Architecture

### Layers

* **Controller (API Layer)**

  * Handles request/response

* **Service Layer**

  * Business logic

* **Data Access Layer (DAL)**

  * DB interaction

---

## 3. DTOs (Data Transfer Objects)

### Purpose

* Separate internal DB schema from API contract

### Benefits

* Flexibility in schema changes
* Security (hide internal fields)

Example:

* DB: user_id, password
* API: user_id only

---

## 4. Request Lifecycle (Detailed)

1. Validate input
2. Authenticate user
3. Authorize access
4. Execute business logic
5. Fetch/update DB
6. Transform to DTO
7. Return response

---

## 5. Sync vs Async Processing

### Synchronous Flow

* Immediate response
* Used for:

  * Reads
  * Small writes

---

### Asynchronous Flow

* Use queues/events

Flow:

* API → Queue → Worker → DB

### When to Use Async

* Long-running tasks
* High throughput systems

---

## 6. CQRS (Command Query Responsibility Segregation)

### Idea

* Separate:

  * **Writes (Commands)**
  * **Reads (Queries)**

---

### Why?

* Different scaling needs
* Read-heavy systems benefit

---

### Implementation

* Write DB (normalized)
* Read DB (denormalized, optimized for queries)

---

## 7. Data Fetching Strategies

### Avoid Over-fetching

* Don’t return unnecessary data

### Avoid Under-fetching

* Don’t force multiple API calls

---

### Techniques

* Aggregation at service layer
* Include related data (`?include=orders`)
* Batch APIs

---

## 8. Consistency Models

### Strong Consistency

* Immediate correctness

### Eventual Consistency

* Data becomes consistent over time

Used with:

* Async processing
* Distributed systems

---

## 9. Caching Integration

### Flow

* API → Cache → DB fallback

### Strategy

* Read-through cache
* Write-through / write-back

---

## 10. Common Interview Mistakes

* Mixing API models with DB schema
* Ignoring async processing
* Not separating read/write concerns
* Over-fetching or under-fetching data
* No clear request flow

---

## TL;DR

* Clean flow: **API → Service → DB**
* Use **DTOs to decouple API & DB**
* Apply **CQRS for scale**
* Use **async processing for heavy work**
* Optimize data fetching carefully
