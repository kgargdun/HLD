# API Design Fundamentals

## 1. REST Basics

### Resource-Oriented Design

* APIs should represent **resources (nouns)**, not actions
* Examples:

  * `/users`
  * `/orders`
  * `/movies`

---

### HTTP Methods

| Method | Purpose          | Idempotent |
| ------ | ---------------- | ---------- |
| GET    | Fetch data       | ✅          |
| POST   | Create resource  | ❌          |
| PUT    | Replace resource | ✅          |
| PATCH  | Partial update   | ❌          |
| DELETE | Remove resource  | ✅          |

---

### HTTP Status Codes

| Code | Meaning               |
| ---- | --------------------- |
| 200  | OK                    |
| 201  | Created               |
| 400  | Bad Request           |
| 401  | Unauthorized          |
| 404  | Not Found             |
| 500  | Internal Server Error |

---

## 2. API Structure

### URL Design

* Use nouns, not verbs
* Keep hierarchy clear

Examples:

* `/users/{id}`
* `/users/{id}/orders`
* `/movies/{id}/shows`

---

### Request / Response Format

#### Request (JSON)

```json
{
  "name": "Kaustubh",
  "email": "test@email.com"
}
```

#### Response (JSON)

```json
{
  "user_id": 123,
  "name": "Kaustubh"
}
```

---

### Pagination (Mandatory)

#### Why?

* Avoid returning large datasets

#### Common Patterns

* Offset-based:

  * `/orders?page=1&limit=20`
* Cursor-based (preferred at scale)

---

### Filtering & Sorting

* `/orders?status=completed`
* `/movies?sort=rating_desc`

---

## 3. Core Principles

### Statelessness

* Server does not store client session
* Each request contains all required info

---

### Idempotency

* Same request → same result

Examples:

* PUT `/users/1` → always same state
* DELETE `/users/1` → safe to retry

---

### Versioning

#### Why?

* Avoid breaking existing clients

#### Methods

* URL versioning:

  * `/v1/users`
* Header-based (less common in interviews)

---

## 4. API Design Best Practices

### Consistency

* Same naming conventions
* Same response structure

---

### Error Handling

Standard format:

```json
{
  "error": {
    "code": "INVALID_INPUT",
    "message": "Email is required"
  }
}
```

---

### Naming Conventions

* Use plural nouns (`/users`, not `/user`)
* Use lowercase + hyphens if needed

---

## 5. Common Interview Mistakes

* Using verbs in endpoints (`/getUsers`)
* Ignoring pagination
* Not handling errors properly
* Breaking idempotency rules
* No versioning strategy

---

## TL;DR

* Design around **resources (nouns)**
* Use correct **HTTP methods + status codes**
* Always include **pagination, filtering, versioning**
* Ensure **stateless + idempotent APIs**
