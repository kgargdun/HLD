# Schema Design

## 1. Relational Schema Design

### Tables & Columns

* Define clear entities → map to tables
* Use appropriate data types (INT, VARCHAR, TIMESTAMP)

### Keys

* **Primary Key (PK)**: Unique row identifier
* **Foreign Key (FK)**: Maintains relationships

Example:

* Users(user_id PK)
* Orders(order_id PK, user_id FK)

---

### Constraints

* **NOT NULL** → mandatory fields
* **UNIQUE** → avoid duplicates
* **CHECK** → enforce conditions

Purpose:

* Maintain data integrity at DB level

---

## 2. NoSQL Schema Design

### Embedding vs Referencing

#### Embedding (Denormalized)

* Store related data together
* Example: User document contains orders

**Pros**

* Fast reads
* No joins

**Cons**

* Data duplication
* Hard updates

---

#### Referencing (Normalized)

* Store relations via IDs

**Pros**

* Cleaner updates
* Less duplication

**Cons**

* Multiple queries needed

---

### When to Use What

* **Embed** → read-heavy, tightly coupled data
* **Reference** → large or frequently updated data

---

## 3. Partition Key Design (Critical)

### What is Partition Key?

* Determines data distribution across nodes

### Good Partition Key

* High cardinality
* Even distribution

### Bad Partition Key

* Causes **hot partitions** (all traffic to one node)

Example:

* Bad → country = "India"
* Good → user_id (high spread)

---

## 4. Scaling-Aware Schema Design

### Sharding (Horizontal Partitioning)

* Split data across multiple machines
* Based on shard/partition key

### Goals

* Distribute load
* Enable horizontal scaling

---

### Avoiding Hotspots

* Avoid skewed keys
* Use hashing if needed

---

### Denormalization for Scale

* Precompute joins
* Store redundant data

Example:

* Store user_name inside Orders table

---

## 5. Read vs Write Tradeoffs

| Design Choice | Impact                      |
| ------------- | --------------------------- |
| Normalized    | Slower reads, faster writes |
| Denormalized  | Faster reads, slower writes |

---

## 6. Schema Evolution

### Challenges

* Changing schema in production

### Strategies

* Backward compatibility
* Additive changes (avoid breaking changes)

---

## 7. Common Interview Mistakes

* Ignoring partition key design
* Using relational mindset in NoSQL
* Overusing joins in high-scale systems
* Not designing for scaling upfront

---

## TL;DR

* Relational → structure + constraints
* NoSQL → access pattern–driven
* Partition key = **most critical decision**
* Denormalize when scaling demands it
