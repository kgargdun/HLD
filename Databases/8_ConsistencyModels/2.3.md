# 🌐 2.3 CONSISTENCY MODELS (STRONG vs EVENTUAL + TUNABLE)

---

## 📘 Definition (textbook-style)

A **consistency model** defines the guarantees a database provides about how and when updates become visible to different nodes and clients in a distributed system.

---

## 💡 Intuition (Why it exists)

In distributed systems:

* Data is replicated across nodes
* Updates don’t reach all nodes instantly

👉 So:

> “When I read data, will I always see the latest value?”

---

## 🧩 Core Problem

After a write:

* Some nodes updated
* Some not yet

👉 What should reads return?

* Latest value?
* Possibly stale value?

---

## 🧩 Types of Consistency Models

---

# 1. 🔒 STRONG CONSISTENCY

---

## 📘 Definition

Every read returns the **most recent committed value**

---

## 💡 Behavior

* Write completes → all future reads see it

---

### 🧠 Example

```id="hsv3v6"
Write: balance = 100 → 50  
Read → always returns 50
```

---

### ✅ Pros

* Simple to reason
* No stale data

---

### ❌ Cons

* High latency
* Requires coordination
* Lower availability

---

### 🌍 Real-world Mapping

* PostgreSQL
* MySQL

---

# 2. 🌊 EVENTUAL CONSISTENCY

---

## 📘 Definition

If no new updates occur, all replicas will **eventually converge** to the same value.

---

## 💡 Behavior

* Immediately after write → some reads may be stale
* Eventually → all nodes consistent

---

### 🧠 Example

```id="21klp8"
Write: status = shipped  
Read → may still show "processing" briefly
```

---

### ✅ Pros

* High availability
* Low latency
* Scales well

---

### ❌ Cons

* Stale reads
* Harder to reason

---

### 🌍 Real-world Mapping

* Cassandra
* DynamoDB

---

# 3. ⚖️ TUNABLE CONSISTENCY (VERY IMPORTANT)

---

## 📘 Definition

System allows controlling consistency level per operation.

---

## 💡 How it works

Use quorum-based approach:

* Write to W nodes
* Read from R nodes
* Total replicas = N

---

### 🧠 Rule

> If **R + W > N → strong consistency**

---

### 💡 Example

* N = 3
* W = 2, R = 2 → consistent
* W = 1, R = 1 → faster but weaker

---

### 🌍 Real-world Mapping

* Cassandra
* DynamoDB

---

## ⚖️ Comparison (VERY IMPORTANT)

| Feature        | Strong Consistency | Eventual Consistency |
| -------------- | ------------------ | -------------------- |
| Data freshness | Always latest      | May be stale         |
| Latency        | Higher             | Lower                |
| Availability   | Lower              | Higher               |
| Complexity     | Simpler            | Complex handling     |

---

## 🎯 When to use what (Decision Thinking)

---

### Use Strong Consistency when:

* Financial systems
* Inventory management
* Critical correctness required

---

### Use Eventual Consistency when:

* Social feeds
* Notifications
* Analytics systems

---

### Use Tunable Consistency when:

* Need flexibility
* Balance performance vs correctness

---

## 🧠 Key Insight (INTERVIEW GOLD)

> “Consistency models define the trade-off between correctness and availability, with most large-scale systems using eventual or tunable consistency for better performance.”

---

## 🗣️ Interview One-liner

> “Consistency models determine how up-to-date data reads are, ranging from strong consistency for correctness to eventual consistency for scalability, with some systems offering tunable guarantees.”

---

## 🚨 Red Flags / Mistakes

* ❌ Assuming all systems are strongly consistent
* ❌ Ignoring stale reads in distributed systems
* ❌ Using eventual consistency for critical data blindly
* ❌ Not understanding quorum logic
* ❌ Confusing consistency with isolation

---

## 🔥 Advanced Insight (VERY IMPORTANT)

👉 Even “strong consistency” systems:

* May degrade under partition
* Trade availability (CAP theorem)

---

## ✍️ Quick Mental Model

> Strong → correct but slow
> Eventual → fast but stale

---
