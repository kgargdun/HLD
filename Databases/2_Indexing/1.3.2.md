# ⚙️ 1.3.2 LSM TREES (WRITE-OPTIMIZED INDEX)

---

## 📘 Definition (textbook-style)

An **LSM Tree (Log-Structured Merge Tree)** is a write-optimized data structure that buffers writes in memory and periodically merges them into sorted files on disk.

---

## 💡 Intuition (Why it exists)

B+ Trees:

* Update data **in-place on disk** ❌
* Causes **random disk I/O** → slow for heavy writes

👉 Problem:

> “How do we make writes fast on disk?”

---

### 💡 Core idea

> “Write sequentially, avoid random disk writes”

---

## 🧩 Core Components

---

## 1. MemTable (In-memory write buffer)

---

### 📘 Definition

An in-memory structure (usually sorted) where writes are first stored.

---

### 💡 Flow

* Write → goes to MemTable (RAM)
* Fast, no disk access

---

## 2. SSTables (Sorted String Tables)

---

### 📘 Definition

Immutable, sorted files stored on disk.

---

### 💡 Key properties

* Written sequentially
* Never modified (immutable)

---

## 3. WAL (Write-Ahead Log)

---

### 📘 Purpose

* Ensures durability

---

### 💡 Flow

* Write → WAL (disk) → MemTable

👉 So crash doesn’t lose data

---

## 🔄 Write Path (IMPORTANT)

---

### ✍️ Step-by-step

1. Write goes to WAL (disk)
2. Write goes to MemTable (RAM)
3. MemTable fills up
4. Flushed to disk as SSTable (sequential write)

---

### 🧠 Key Insight

> Writes are **sequential + batched → very fast**

---

## 🔄 Read Path (MORE COMPLEX)

---

### 🔍 Steps

To read a key:

1. Check MemTable
2. Check multiple SSTables
3. Merge results if needed

---

### ⚠️ Problem

* Data spread across many files
* Slower reads than B+ Trees

---

## 🔄 Compaction (CRITICAL)

---

### 📘 Definition

Process of merging multiple SSTables into fewer, optimized ones.

---

### 💡 Why needed?

* Remove duplicates
* Reduce number of files
* Improve read performance

---

### ⚠️ Trade-off

* Compaction is expensive (background work)

---

## 🌍 Real-world Mapping

---

* Cassandra
* RocksDB
* DynamoDB (internally similar concepts)

---

## ⚖️ LSM vs B+ Tree (VERY IMPORTANT)

| Feature  | B+ Tree           | LSM Tree          |
| -------- | ----------------- | ----------------- |
| Writes   | Slow (random I/O) | Fast (sequential) |
| Reads    | Fast              | Slower            |
| Storage  | In-place updates  | Immutable files   |
| Use case | Read-heavy        | Write-heavy       |

---

## 🎯 When to use vs not use

---

### Use LSM Trees when:

* Write-heavy workloads
* High ingestion rate
* Append-heavy systems

#### Examples:

* Logs
* Metrics
* Messaging

---

### Avoid LSM Trees when:

* Low-latency reads required
* Heavy range queries
* Read-heavy workloads

---

## 🧠 Key Insight (INTERVIEW GOLD)

> “LSM Trees trade read performance for extremely fast sequential writes by batching and merging data on disk.”

---

## 🗣️ Interview One-liner

> “LSM Trees optimize writes by buffering them in memory and flushing sequentially to disk, making them ideal for high-write systems but with more complex read paths.”

---

## 🚨 Red Flags / Mistakes

* ❌ Saying LSM is always better (depends on workload)
* ❌ Ignoring compaction cost
* ❌ Using LSM for read-heavy systems
* ❌ Not mentioning WAL (durability)
* ❌ Forgetting read amplification issue

---

## ✍️ Quick Mental Model

> Write → Memory → Sequential disk
> Read → Check multiple places

---

## 🔥 Interview Insight (VERY IMPORTANT)

👉 Most modern systems:

* Use LSM for **writes**
* Add caching/indexing to fix reads
