# ⚙️ 1.3 INDEXING (INTRO + WHY INDEXES)

---

## 📘 Definition (textbook-style)

An **index** is a data structure that improves the speed of data retrieval operations on a database table by providing a faster lookup path than scanning the entire dataset.

---

## 💡 Intuition (Why it exists)

Without an index:

* DB must scan entire table → **O(n)**
* Slow as data grows

With an index:

* DB uses a shortcut → **O(log n)** or better

👉 Real-world analogy:

> Like a book index — jump directly to the page instead of reading everything

---

## 🧩 Problem Without Index

---

### 🔴 Full Table Scan

```sql
SELECT * FROM users WHERE user_id = 100;
```

Without index:

* Check every row ❌
* Time grows linearly

---

### 🟢 Indexed Lookup

With index on `user_id`:

* Jump directly to location ✅
* Much faster

---

## ⚡ What Index Actually Does

---

### 💡 Concept

Index stores:

👉 (key → pointer to actual data)

Example:

```
100 → location of row on disk
200 → location of row on disk
```

---

### 🧠 Key Insight

> Index is separate from actual data — it’s an auxiliary structure

---

## 🔄 Types of Operations Improved

---

### ✅ Fast with Index

* Point lookup (`WHERE id = x`)
* Range queries (`BETWEEN`)
* Sorting (`ORDER BY`)
* Joins (on indexed columns)

---

### ❌ Not helped (or less helpful)

* Full table scans (if most rows needed)
* Small tables (index overhead > benefit)

---

## ⚖️ Trade-offs (VERY IMPORTANT)

---

### ✅ Benefits

* Faster reads
* Efficient filtering
* Better query performance

---

### ❌ Costs

* Slower writes (index must be updated)
* Extra storage
* Maintenance overhead

---

### ⚠️ Critical Trade-off

> Every index improves reads but hurts writes

---

## 🎯 When to use vs not use

---

### Use index when:

* Column is frequently queried
* High selectivity (many unique values)
* Used in joins or filters

---

### Avoid index when:

* Column rarely queried
* Low cardinality (e.g., gender: M/F)
* Write-heavy system

---

## 🌍 Real-world Mapping

---

* PostgreSQL → B+ tree indexes
* MySQL → B+ tree indexes
* Cassandra → limited indexing (write-heavy focus)

---

## 🧠 Key Insight (INTERVIEW GOLD)

> “Indexes trade write performance and storage for faster reads.”

---

## 🗣️ Interview One-liner

> “Indexes provide a faster lookup path to data by avoiding full table scans, typically reducing query complexity from O(n) to O(log n), at the cost of slower writes and additional storage.”

---

## 🚨 Red Flags / Mistakes

* ❌ Adding indexes everywhere (kills write performance)
* ❌ Indexing low-cardinality columns
* ❌ Ignoring index maintenance cost
* ❌ Assuming index always used (query planner decides)
* ❌ Not indexing join keys

---

## ✍️ Quick Mental Model

> Index = Shortcut to data
>
> No index → scan everything
> Index → jump directly
