# ⚙️ 1.3.1 B+ TREES (PRIMARY INDEX STRUCTURE)

---

## 📘 Definition (textbook-style)

A **B+ Tree** is a self-balancing tree data structure used in databases to store sorted data and allow efficient insertion, deletion, and search operations in **O(log n)** time.

---

## 💡 Intuition (Why it exists)

We need an index that:

* Works well on **disk (not RAM)**
* Minimizes disk I/O
* Supports:

  * Fast lookups
  * Efficient range queries

👉 B+ Tree is optimized for:

> “Read as few disk pages as possible”

---

## 🧩 Structure

---

### 🌳 Key properties

* Tree with multiple children per node (high branching factor)
* Data stored in **sorted order**
* **All actual data pointers are in leaf nodes**
* Leaf nodes are **linked (important for range queries)**

---

### 💡 Simplified view

```
        [10 | 20]
       /    |    \
   [1..9] [10..19] [20..]
```

* Internal nodes → guide traversal
* Leaf nodes → actual data references

---

## ⚡ Why B+ Trees are used (DISK OPTIMIZATION)

---

### 🧠 Key idea: Minimize disk reads

* Each node ≈ one disk page
* High branching factor → shallow tree

👉 Example:

* Millions of rows → tree height ~ 3–4

So lookup = **3–4 disk reads only**

---

## 🔄 How operations work

---

### 🔍 Search

1. Start at root
2. Follow correct child pointer
3. Reach leaf
4. Get data pointer

👉 Complexity: O(log n)

---

### ➕ Insert

* Insert into correct leaf
* If full → split node
* May propagate upwards

---

### ❌ Delete

* Remove entry
* Merge or redistribute if needed

---

## 🚀 Range Queries (BIG ADVANTAGE)

---

### 💡 Example

```sql
SELECT * FROM orders WHERE price BETWEEN 100 AND 200;
```

---

### ⚡ Why efficient?

* Leaf nodes are **linked sequentially**

👉 So:

* Find start → scan sequentially

---

### 🧠 Key Insight

> B+ Trees are optimized for both point lookups AND range scans

---

## 🌍 Real-world Mapping

---

* PostgreSQL
* MySQL

👉 Almost all relational DB indexes = B+ Trees

---

## ⚖️ Trade-offs

---

### ✅ Pros

* Efficient disk access
* Supports range queries
* Balanced → predictable performance
* Good for read-heavy systems

---

### ❌ Cons

* Slower writes (node splits)
* Random I/O heavy
* Not optimal for write-heavy workloads

---

## 🎯 When to use vs not use

---

### Use B+ Trees when:

* Need range queries
* Moderate write load
* OLTP workloads

---

### Avoid B+ Trees when:

* Very high write throughput required
* Sequential writes preferred

👉 Use LSM trees instead → (next topic)

---

## 🧠 Key Insight (INTERVIEW GOLD)

> “B+ Trees optimize for disk access by keeping the tree shallow and storing data in sorted order, making them ideal for both point lookups and range queries.”

---

## 🗣️ Interview One-liner

> “B+ Trees are the default indexing structure in databases because they minimize disk I/O and efficiently support both point and range queries.”

---

## 🚨 Red Flags / Mistakes

* ❌ Saying B-tree and B+ tree are same (they differ)
* ❌ Ignoring disk-page optimization aspect
* ❌ Not mentioning linked leaf nodes
* ❌ Using B+ tree logic for write-heavy systems blindly

---

## ✍️ Quick Mental Model

> Wide tree (not deep)
> → Few disk reads
> → Sorted leaves → fast range scan
