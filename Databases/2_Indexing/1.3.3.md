# ⚙️ 1.3.3 SECONDARY INDEXES (VERY IMPORTANT)

---

## 📘 Definition (textbook-style)

A **secondary index** is an index created on a column (or set of columns) that is **not the primary key**, allowing efficient queries based on non-primary attributes.

---

## 💡 Intuition (Why it exists)

Primary key index:

* Only helps for `WHERE id = X`

👉 But real queries are like:

```sql
SELECT * FROM users WHERE email = 'x';
SELECT * FROM orders WHERE status = 'shipped';
```

👉 So we need:

> “Indexes on other columns besides primary key”

---

## 🧩 Types of Indexes (Conceptual)

---

### 1. Primary Index

* Built on primary key
* Usually clustered (data stored in order)

---

### 2. Secondary Index

* Built on non-primary columns
* Separate structure from main data

---

## ⚡ How Secondary Index Works

---

### 💡 Concept

Instead of:

```
email → full row
```

It stores:

```
email → pointer to primary key (or row location)
```

---

### 🔄 Query flow

1. Use secondary index → find matching entries
2. Get primary key / row pointer
3. Fetch actual row

---

### 🧠 Key Insight

> Secondary index = extra lookup step

---

## 📌 Example

---

### Without index

```sql
SELECT * FROM users WHERE email = 'a@gmail.com';
```

→ Full table scan ❌

---

### With secondary index on email

→ Index lookup → row pointer → fetch row ✅

---

## ⚖️ Trade-offs (VERY IMPORTANT)

---

### ✅ Pros

* Faster queries on non-primary fields
* Enables flexible querying
* Improves joins

---

### ❌ Cons (CRITICAL)

---

### 1. Write Amplification

* Insert/update → update ALL indexes

👉 More indexes → slower writes

---

### 2. Storage Overhead

* Each index consumes memory + disk

---

### 3. Read Amplification (in some systems)

* Extra lookup (index → data)

---

### 4. Consistency Complexity (in distributed systems)

* Keeping index in sync is hard

---

## 🌍 Real-world Mapping

---

### Relational DBs

* PostgreSQL → rich indexing
* MySQL

---

### NoSQL systems

* Cassandra → limited support
* DynamoDB → requires explicit design

👉 Reason:

> Secondary indexes are expensive at scale

---

## ⚖️ Secondary Index in B+ Tree vs LSM (INTERVIEW GOLD)

---

### In B+ Tree systems

* Efficient
* Direct lookup

---

### In LSM systems

* More complex
* Needs separate index structures
* High write amplification

---

## 🎯 When to use vs not use

---

### Use secondary index when:

* Column frequently queried
* High selectivity (many unique values)
* Needed for joins

---

### Avoid secondary index when:

* Write-heavy system
* Column rarely queried
* Low cardinality (e.g., status = active/inactive)

---

## 🧠 Key Insight (VERY IMPORTANT)

> “Indexes are not free — every additional index increases write cost and system complexity.”

---

## 🗣️ Interview One-liner

> “Secondary indexes enable efficient querying on non-primary fields but introduce write amplification and consistency challenges, especially in distributed systems.”

---

## 🚨 Red Flags / Mistakes

* ❌ Adding indexes without considering write cost
* ❌ Indexing low-cardinality fields
* ❌ Assuming indexes scale infinitely
* ❌ Ignoring distributed index maintenance cost
* ❌ Overusing secondary indexes in NoSQL

---

## 🔥 Advanced Insight (HIGH VALUE)

In large-scale systems:

👉 Instead of secondary indexes:

* Use **denormalization**
* Maintain **materialized views manually**

Example:

* Store data by multiple access patterns

---

## ✍️ Quick Mental Model

> Secondary Index = Extra lookup path
>
> Faster reads ❗
> Slower writes ❗
