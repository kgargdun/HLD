# 🧠 0.2 WORKLOAD TYPES

---

## 📘 Definition (textbook-style)

A **workload** refers to the pattern of operations (reads, writes, queries) that a database system must handle, including their frequency, complexity, and latency requirements.

---

## 💡 Intuition (Why it exists)

Not all systems use databases the same way:

* Banking → small, consistent transactions
* Analytics → huge scans, aggregations
* Social media → massive reads + writes

👉 So the question becomes:

> “What kind of work will my database do most of the time?”

This directly drives:

* DB choice
* Indexing strategy
* Scaling approach

---

## 🧩 Core Workload Types

---

### 1. OLTP (Online Transaction Processing)

#### 📘 Definition

Systems that handle **large number of small, fast, transactional operations**.

#### 💡 Characteristics

* High QPS (queries per second)
* Small reads/writes
* Strong consistency required
* Low latency critical

#### 🧠 Examples

* Payments
* Order systems
* User login

#### 🌍 Real-world Mapping

* PostgreSQL
* MySQL
* DynamoDB

---

### 2. OLAP (Online Analytical Processing)

#### 📘 Definition

Systems designed for **complex queries over large datasets**, typically for analytics.

#### 💡 Characteristics

* Large scans (millions/billions of rows)
* Aggregations (SUM, COUNT, AVG)
* Read-heavy
* Latency less critical (seconds OK)

#### 🧠 Examples

* Business dashboards
* Data analysis
* Reporting systems

#### 🌍 Real-world Mapping

* BigQuery
* Redshift

---

## ⚖️ OLTP vs OLAP (IMPORTANT)

| Feature    | OLTP            | OLAP         |
| ---------- | --------------- | ------------ |
| Query type | Simple          | Complex      |
| Data size  | Small per query | Huge scans   |
| Latency    | ms              | sec/min      |
| Writes     | Frequent        | Rare/batched |
| Schema     | Normalized      | Denormalized |

---

## 🔄 3. Read-heavy vs Write-heavy Workloads

---

### 📖 Read-heavy Systems

#### 💡 Characteristics

* Reads >> Writes
* Need fast access
* Often cacheable

#### 🧠 Examples

* News feed
* Product catalog

#### 🚀 Optimization

* Caching → Redis
* Read replicas

---

### ✍️ Write-heavy Systems

#### 💡 Characteristics

* Writes >> Reads
* High ingestion rate
* Append-heavy

#### 🧠 Examples

* Logging systems
* Chat messages
* Metrics ingestion

#### 🚀 Optimization

* LSM-tree DBs → Cassandra
* Batch writes

---

## ⚡ 4. Latency-sensitive vs Throughput-oriented

---

### ⏱️ Latency-sensitive

* Each request must be fast (p99 matters)
* Example:

  * Payments
  * API calls

---

### 📦 Throughput-oriented

* Total work per second matters
* Individual request latency less important

#### Example:

* Data pipelines
* Analytics jobs

---

## ⚖️ Trade-offs

| Optimization for   | Trade-off           |
| ------------------ | ------------------- |
| Low latency        | Reduced throughput  |
| High throughput    | Increased latency   |
| Strong consistency | Higher latency      |
| High availability  | Possible stale data |

---

## 🎯 When to use what (Decision thinking)

---

### Use OLTP systems when:

* Need transactions
* Need low latency
* User-facing operations

---

### Use OLAP systems when:

* Running analytics
* Aggregating large datasets
* Reporting / dashboards

---

### Use hybrid carefully:

* Mixing OLTP + OLAP → performance issues
  👉 Solution:
* Separate systems (ETL pipelines)

---

## 🗣️ Interview One-liner

> “Workload type defines database design — OLTP systems optimize for low-latency transactions, while OLAP systems optimize for large-scale analytical queries.”

---

## 🚨 Red Flags / Mistakes

* ❌ Using OLTP DB for analytics (slow, expensive)
* ❌ Running heavy queries on production DB
* ❌ Ignoring read/write ratio while designing schema
* ❌ No caching in read-heavy systems
* ❌ Not batching writes in write-heavy systems
