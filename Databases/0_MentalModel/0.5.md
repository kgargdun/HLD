# 🧠 0.5 DATA MODELING BASICS

---

## 📘 Definition (textbook-style)

**Data modeling** is the process of structuring data — defining entities, relationships, and constraints — to efficiently store and retrieve data in a database.

---

## 💡 Intuition (Why it exists)

Database performance is not just about the DB engine — it’s heavily driven by **how data is structured**.

👉 Same database + bad modeling = poor performance
👉 Good modeling = fewer joins, faster queries, better scalability

> “Design schema based on how data is accessed, not how it looks logically.”

---

## 🧩 Core Concepts

---

### 1. Entities

#### 📘 Definition

An **entity** represents a real-world object.

#### 💡 Examples

* User
* Order
* Product

---

### 2. Attributes

#### 📘 Definition

Properties of an entity.

#### 💡 Example

User:

* user_id
* name
* email

---

### 3. Relationships

#### 📘 Definition

How entities are connected.

#### 💡 Types

* One-to-One
* One-to-Many
* Many-to-Many

---

### 4. Schema

#### 📘 Definition

Structure of the database (tables, fields, types)

---

## ⚖️ Normalization vs Denormalization (VERY IMPORTANT)

---

### 🔹 Normalization

#### 📘 Definition

Breaking data into multiple tables to reduce redundancy.

#### 💡 Example

* Users table
* Orders table (reference user_id)

#### ✅ Pros

* No data duplication
* Strong consistency
* Easier updates

#### ❌ Cons

* Requires joins
* Slower queries at scale

#### 🌍 Mapping

* PostgreSQL
* MySQL

---

### 🔹 Denormalization

#### 📘 Definition

Storing redundant data to optimize reads.

#### 💡 Example

* Orders table includes user_name directly

#### ✅ Pros

* Faster reads
* Fewer joins
* Better for scale

#### ❌ Cons

* Data duplication
* Harder updates
* Risk of inconsistency

#### 🌍 Mapping

* Cassandra
* DynamoDB

---

## ⚖️ Decision: Normalize vs Denormalize

| Scenario                  | Choice      |
| ------------------------- | ----------- |
| Strong consistency needed | Normalize   |
| Read-heavy system         | Denormalize |
| Complex joins required    | Normalize   |
| Large-scale system        | Denormalize |

---

## 🧠 Key Design Principle (INTERVIEW GOLD)

> “In scalable systems, we design schema based on queries, not relationships.”

---

## 🎯 When to use what

---

### Use normalization when:

* Data integrity is critical
* Frequent updates
* Complex relationships

---

### Use denormalization when:

* Read-heavy workload
* High scale
* Joins are expensive

---

## 🌍 Real-world Mapping

---

### Relational (normalized)

* PostgreSQL
* Banking, transactions

---

### NoSQL (denormalized)

* Cassandra
* DynamoDB
* Social feeds, logs

---

## 🗣️ Interview One-liner

> “Data modeling is about structuring data based on access patterns — normalized for consistency, and denormalized for scalability and performance.”

---

## 🚨 Red Flags / Mistakes

* ❌ Over-normalizing in high-scale systems
* ❌ Using joins in hot paths
* ❌ Not modeling based on queries
* ❌ Ignoring update complexity in denormalization
* ❌ Copying relational design into NoSQL blindly

---

## ✍️ Quick Mental Model

> Normalize → correctness
> Denormalize → performance

---

## 🧠 Interview Insight (VERY IMPORTANT)

Most large-scale systems:

👉 Start normalized
👉 Move to denormalized as scale increases
