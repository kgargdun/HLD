# DATABASE TYPES & SYSTEM MAPPING (LEVEL 3)

---

## 1. RELATIONAL DATABASES (SQL)

### Definition
Relational databases store data in structured tables with predefined schemas and support ACID transactions.

### Core Characteristics
- Schema-based (strict structure)
- Supports joins
- Strong consistency (ACID)
- Uses B+ tree indexes

### Strengths
- Data integrity
- Complex queries (joins, aggregations)
- Transaction support

### Weaknesses
- Hard to scale horizontally
- Joins expensive at scale

### When to Use
- Financial systems
- Inventory management
- Systems requiring strong consistency

### When NOT to Use
- Massive scale with high write throughput
- Flexible schema requirements

### Real Systems
- PostgreSQL
- MySQL

---

## 2. KEY-VALUE STORES

### Definition
Stores data as key → value pairs with simple GET/PUT operations.

### Core Characteristics
- No schema
- Extremely fast lookups
- Typically in-memory or LSM-based

### Strengths
- Very low latency
- Horizontally scalable
- Simple design

### Weaknesses
- No joins
- Limited querying capability

### When to Use
- Caching
- Session storage
- Simple lookup systems

### When NOT to Use
- Complex queries needed
- Relationships between data

### Real Systems
- Redis
- DynamoDB

---

## 3. DOCUMENT DATABASES

### Definition
Stores data as semi-structured documents (JSON/BSON).

### Core Characteristics
- Flexible schema
- Nested data
- Supports indexing on fields

### Strengths
- Schema flexibility
- Good for hierarchical data
- Easier than joins

### Weaknesses
- Limited transactions (compared to SQL)
- Can become inconsistent if poorly designed

### When to Use
- User profiles
- Content management
- Evolving schemas

### When NOT to Use
- Strong transactional systems
- Complex joins required

### Real Systems
- MongoDB

---

## 4. WIDE-COLUMN STORES

### Definition
Stores data in column families, optimized for large-scale distributed systems.

### Core Characteristics
- LSM-tree based
- High write throughput
- Distributed by design

### Strengths
- Massive scalability
- High write performance
- Fault tolerant

### Weaknesses
- Limited querying
- Eventual consistency

### When to Use
- Logging systems
- Time-series data
- Write-heavy workloads

### When NOT to Use
- Strong consistency required
- Complex queries

### Real Systems
- Cassandra

---

## 5. SEARCH ENGINES

### Definition
Specialized systems for full-text search using inverted indexes.

### Core Characteristics
- Tokenization
- Ranking (TF-IDF, BM25)
- Fuzzy matching

### Strengths
- Full-text search
- Relevance ranking
- Fast keyword queries

### Weaknesses
- Not a primary database
- Eventual consistency

### When to Use
- Search features
- Logs search
- Product search

### When NOT to Use
- Transactional systems
- Source of truth

### Real Systems
- Elasticsearch

---

## 6. TIME-SERIES DATABASES

### Definition
Optimized for storing and querying time-stamped data.

### Core Characteristics
- Time-based indexing
- High ingestion rate
- Compression optimized

### Strengths
- Efficient time queries
- High write throughput

### Weaknesses
- Limited general-purpose usage

### When to Use
- Metrics
- Monitoring systems
- IoT data

### Real Systems
- InfluxDB

---

## 7. DATA WAREHOUSE (OLAP)

### Definition
Designed for analytical queries over large datasets.

### Core Characteristics
- Columnar storage
- Batch processing
- Optimized for scans

### Strengths
- Fast aggregations
- Handles massive data

### Weaknesses
- Not for real-time transactions

### When to Use
- Analytics
- Reporting
- BI dashboards

### Real Systems
- BigQuery
- Redshift

---

## 🧠 KEY INSIGHT (INTERVIEW GOLD)

No single database solves all problems.

👉 Real systems use:
- SQL (core data)
- NoSQL (scale)
- Cache (performance)
- Search (query)

---

## 🔗 COMMON ARCHITECTURE PATTERN

User Request →
- Cache (Redis)
- DB (PostgreSQL)
- Search (Elasticsearch)
- Analytics (BigQuery)

---

## 🚨 COMMON MISTAKES

- Using one DB for everything
- Using SQL for massive scale blindly
- Using NoSQL without understanding trade-offs
- Using search DB as primary DB
