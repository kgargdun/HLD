# ⚙️ 1.4 QUERY EXECUTION (HOW DATABASES RUN QUERIES)

---

## 📘 Definition (textbook-style)

**Query execution** is the process by which a database parses, plans, optimizes, and executes a query to retrieve or modify data.

---

## 💡 Intuition (Why it exists)

When you write:

```sql
SELECT * FROM users WHERE age > 25;
```

👉 Database doesn’t “just run it”

It goes through multiple steps to decide:

> “What is the fastest way to get this data?”

---

## 🧩 Query Execution Pipeline

---

## 1. Parsing

---

### 📘 Definition

Convert SQL query into an internal representation (parse tree)

---

### 💡 What happens

* Syntax check
* Validate query structure

---

### ❌ Example error

```sql
SELEC * FROM users;
```

→ Fails at parsing

---

## 2. Query Planning (Logical Plan)

---

### 📘 Definition

Determine **what operations are needed**

---

### 💡 Example

Query:

```sql
SELECT * FROM users WHERE age > 25;
```

Logical plan:

* Scan table
* Apply filter

---

## 3. Query Optimization (VERY IMPORTANT)

---

### 📘 Definition

Choose the **most efficient execution strategy**

---

### 💡 Why needed?

Same query → multiple ways to execute

---

### 🔍 Example

```sql
SELECT * FROM orders WHERE user_id = 10;
```

Options:

* Full table scan ❌
* Use index on user_id ✅

---

### 🧠 Cost-Based Optimizer

DB evaluates:

* Data size
* Index availability
* Selectivity

👉 Picks cheapest plan

---

### 🧠 Key Insight

> Query performance depends more on execution plan than SQL itself

---

## 4. Execution Engine

---

### 📘 Definition

Actually runs the chosen plan

---

### 🔄 Common operations

---

### 🔹 a) Sequential Scan

* Read entire table
* Used when:

  * No index
  * Large portion needed

---

### 🔹 b) Index Scan

* Use index → fetch rows

---

### 🔹 c) Join Algorithms (IMPORTANT)

---

#### 1. Nested Loop Join

* For each row in A → scan B
* Simple but slow

---

#### 2. Hash Join

* Build hash table on one table
* Probe with other

👉 Fast for large datasets

---

#### 3. Merge Join

* Requires sorted data
* Very efficient

---

## ⚖️ Trade-offs

---

| Strategy   | Pros                 | Cons                  |
| ---------- | -------------------- | --------------------- |
| Seq Scan   | Simple               | Slow for large tables |
| Index Scan | Fast lookup          | Random I/O cost       |
| Hash Join  | Fast for large joins | Memory heavy          |
| Merge Join | Very efficient       | Needs sorted input    |

---

## 🌍 Real-world Mapping

---

* PostgreSQL → advanced optimizer
* MySQL

---

## 🎯 When to use vs not use (Design Thinking)

---

### Use indexes when:

* Query filters small subset

---

### Avoid index when:

* Query returns large portion
  👉 Seq scan may be faster

---

### Design joins carefully:

* Large joins → expensive
* Prefer denormalization at scale

---

## 🧠 Key Insight (INTERVIEW GOLD)

> “Databases don’t execute queries directly — they choose an execution plan using a cost-based optimizer, and performance depends heavily on that plan.”

---

## 🗣️ Interview One-liner

> “Query execution involves parsing, planning, optimizing, and executing a query, with the optimizer selecting the most efficient strategy based on data distribution and indexes.”

---

## 🚨 Red Flags / Mistakes

* ❌ Thinking SQL directly maps to execution
* ❌ Ignoring query plan (EXPLAIN is critical)
* ❌ Blindly adding indexes without checking plan
* ❌ Large joins in hot paths
* ❌ Not understanding scan vs index trade-offs

---

## 🔥 Interview Insight (VERY IMPORTANT)

👉 If query is slow:

Don’t change query first ❌
👉 Check execution plan first ✅

---

## ✍️ Quick Mental Model

> SQL → Parse → Plan → Optimize → Execute

---

## 🔗 Quick Interview Mapping

| Problem          | Optimization |
| ---------------- | ------------ |
| Slow query       | Check plan   |
| Large joins      | Denormalize  |
| Full scan        | Add index    |
| Too many indexes | Remove some  |
