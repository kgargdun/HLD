# ⚙️ 1.1 STORAGE ENGINE (DISK vs MEMORY, PAGE, BUFFER POOL)

---

## 📘 Definition (textbook-style)

A **storage engine** is the component of a database responsible for how data is physically stored, retrieved, and managed on disk or in memory.

---

## 💡 Intuition (Why it exists)

At the lowest level:

* Data lives on **disk (slow)**
* Queries need **fast access (memory)**

👉 Storage engine bridges this gap:

> “How do we efficiently read/write disk data while serving fast queries?”

---

## 🧩 Core Components

---

## 1. Disk vs Memory Storage

---

### 💾 Disk Storage

#### 💡 Characteristics

* Persistent (durable)
* Slow (milliseconds latency)
* Requires block-based access

#### 🧠 Key Insight

* Disk reads are expensive → minimize I/O

---

### ⚡ Memory (RAM)

#### 💡 Characteristics

* Very fast (nanoseconds)
* Not durable
* Limited size

---

### ⚖️ Trade-off

| Factor     | Disk       | Memory    |
| ---------- | ---------- | --------- |
| Speed      | Slow       | Fast      |
| Durability | Persistent | Volatile  |
| Cost       | Cheap      | Expensive |

---

### 🌍 Real-world Mapping

* Disk-based DB → PostgreSQL
* In-memory DB → Redis

---

## 2. Page / Block Structure

---

### 📘 Definition

Data on disk is stored in **fixed-size blocks (pages)**, typically 4KB–16KB.

---

### 💡 Why pages?

Disk doesn’t read:

* “1 row” ❌
* It reads:
* “1 block/page” ✅

---

### 🧠 Example

If you query 1 row:

* Entire page is loaded into memory

👉 So:

> DB optimizes for **page-level access, not row-level**

---

### ⚠️ Implication

* Sequential access → fast
* Random access → expensive

---

## 3. Buffer Pool (VERY IMPORTANT)

---

### 📘 Definition

A **buffer pool** is an in-memory cache of disk pages used by the database to reduce disk I/O.

---

### 💡 Intuition

Instead of:

* Reading disk every time ❌

DB does:

* Keep frequently used pages in memory ✅

---

### 🔄 How it works

1. Query comes
2. Check buffer pool:

   * If page present → return (fast)
   * Else → read from disk → store in buffer → return

---

### ⚡ Optimization Strategy

* LRU / LFU caching
* Hot data stays in memory

---

### 🌍 Real-world Mapping

* MySQL (InnoDB buffer pool)
* PostgreSQL (shared buffers)

---

## ⚖️ Trade-offs

| Optimization         | Trade-off                            |
| -------------------- | ------------------------------------ |
| Large buffer pool    | More memory usage                    |
| Small buffer pool    | More disk I/O                        |
| Caching aggressively | Risk of stale data (in some systems) |

---

## 🎯 When to use what (Decision Thinking)

---

### Use disk-heavy systems when:

* Need durability
* Large datasets

---

### Use in-memory systems when:

* Ultra-low latency required
* Data can be rebuilt

---

### Hybrid (most common):

* Disk + buffer pool

---

## 🗣️ Interview One-liner

> “Storage engines optimize disk access using page-based storage and buffer pools to minimize expensive I/O while maintaining durability.”

---

## 🚨 Red Flags / Mistakes

* ❌ Thinking DB reads row-by-row from disk
* ❌ Ignoring disk I/O cost
* ❌ Not tuning buffer pool size
* ❌ Using in-memory DB for critical persistent data
* ❌ Assuming cache = always consistent

---

## ✍️ Quick Mental Model

> Disk (slow, durable)
>
> * Buffer Pool (fast cache)
> * Pages (unit of I/O)

---

## 🧠 Interview Insight (VERY IMPORTANT)

👉 Most DB performance issues are actually:

> Disk I/O problems, not query problems
