# INTERVIEW MAPPING (LEVEL 6)

---

## 🎯 GOAL

This section helps you answer:

> “Which database would you use and why?”

👉 Fast, structured, high-signal answers (no deep dive)

---

## 🧠 CORE STRATEGY

For ANY system:

1. Identify workload pattern  
2. Identify access pattern  
3. Identify consistency needs  
4. Map to DB choice  

---

# 🧩 1. SOCIAL MEDIA (TWITTER / INSTAGRAM)

---

## 💡 Characteristics
- Read-heavy (feeds)
- High write volume (posts)
- Eventually consistent OK

---

## 🧠 DB Choices

- Primary DB → NoSQL (Cassandra)
- Cache → Redis
- Feed storage → denormalized
- Search → Elasticsearch

---

## ⚡ Key Insight

- Fan-out pattern
- Denormalization critical

---

## 🗣️ Interview One-liner

> “I would use a NoSQL DB like Cassandra for scalability, cache feeds in Redis, and denormalize data to avoid joins.”

---

---

# 🧩 2. CHAT SYSTEM (WHATSAPP)

---

## 💡 Characteristics
- Write-heavy
- Ordering important
- Low latency

---

## 🧠 DB Choices

- Messages → Cassandra / DynamoDB
- Cache → Redis
- Metadata → SQL (optional)

---

## ⚡ Key Insight

- Append-only writes
- Partition by user/conversation

---

## 🗣️ Interview One-liner

> “I would use a write-optimized NoSQL store like Cassandra for messages, partitioned by conversation, with Redis for fast access.”

---

---

# 🧩 3. RIDE SHARING (UBER)

---

## 💡 Characteristics
- Real-time updates
- Geo queries
- Strong consistency (for rides)

---

## 🧠 DB Choices

- Core data → SQL (PostgreSQL)
- Geo indexing → specialized index
- Cache → Redis

---

## ⚡ Key Insight

- Geo queries critical
- Consistency important for booking

---

## 🗣️ Interview One-liner

> “I would use PostgreSQL for strong consistency and geo-indexing, combined with Redis for real-time location caching.”

---

---

# 🧩 4. E-COMMERCE (AMAZON)

---

## 💡 Characteristics
- Mix of read-heavy + transactional
- Inventory consistency critical

---

## 🧠 DB Choices

- Orders / payments → SQL
- Product catalog → NoSQL
- Cache → Redis
- Search → Elasticsearch

---

## ⚡ Key Insight

- Hybrid system
- Different DBs for different needs

---

## 🗣️ Interview One-liner

> “I would use SQL for transactions like orders, NoSQL for product catalog, Redis for caching, and Elasticsearch for search.”

---

---

# 🧩 5. LOGGING / METRICS SYSTEM

---

## 💡 Characteristics
- Extremely write-heavy
- Append-only
- High ingestion rate

---

## 🧠 DB Choices

- Storage → Cassandra / InfluxDB
- Analytics → BigQuery

---

## ⚡ Key Insight

- LSM tree systems ideal

---

## 🗣️ Interview One-liner

> “I would use an LSM-based database like Cassandra for high write throughput and a warehouse like BigQuery for analytics.”

---

---

# 🧩 6. SEARCH SYSTEM

---

## 💡 Characteristics
- Full-text search
- Ranking
- Fuzzy matching

---

## 🧠 DB Choices

- Primary DB → SQL/NoSQL
- Search index → Elasticsearch

---

## ⚡ Key Insight

- Search DB is secondary index

---

## 🗣️ Interview One-liner

> “I would store data in a primary DB and index it into Elasticsearch for full-text search capabilities.”

---

---

# 🧩 7. ANALYTICS DASHBOARD

---

## 💡 Characteristics
- Heavy aggregations
- Large scans

---

## 🧠 DB Choices

- OLAP → BigQuery / Redshift

---

## ⚡ Key Insight

- Columnar storage

---

## 🗣️ Interview One-liner

> “I would use a columnar data warehouse like BigQuery for efficient large-scale aggregations.”

# 🧠 MASTER INTERVIEW TEMPLATE

---

## When asked:

“What database would you use?”

---

## Answer structure:

1. “Based on access pattern…”  
2. “Considering workload…”  
3. “For consistency…”  
4. “I would choose…”  

---

## Example

“I would analyze access patterns first — since this is read-heavy with simple lookups, I would use a NoSQL DB like DynamoDB, add Redis for caching, and use replication for scaling reads.”
