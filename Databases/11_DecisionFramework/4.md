# DATABASE DECISION FRAMEWORK (LEVEL 4)

---

## 🎯 GOAL

In interviews, you are NOT judged on knowing databases.

👉 You are judged on:
> “Why did you choose THIS database over others?”

---

## 🧠 CORE PRINCIPLE

> Always choose database based on:
> **Access Patterns + Scale + Consistency Needs**

---

# 🧩 STEP-BY-STEP DECISION FRAMEWORK

---

## 1. ACCESS PATTERN (FIRST FILTER)

---

### Ask:

- Point lookup? → Key-value
- Range queries? → SQL (B+ tree)
- Aggregations? → OLAP
- Full-text search? → Search engine

---

### Mapping

| Pattern          | DB Choice        |
|------------------|-----------------|
| Key lookup       | Redis, DynamoDB |
| Range queries    | PostgreSQL      |
| Aggregations     | BigQuery        |
| Text search      | Elasticsearch   |

---

## 🧠 Insight

> Access pattern eliminates 50% of wrong choices immediately

---

## 2. WORKLOAD TYPE

---

### Ask:

- OLTP → transactional system?
- OLAP → analytics system?

---

### Mapping

| Workload | DB Type |
|----------|--------|
| OLTP     | SQL / NoSQL |
| OLAP     | Column DB |

---

## 🚨 Rule

> NEVER mix OLTP + OLAP in same DB at scale

---

## 3. READ vs WRITE RATIO

---

### Ask:

- Read-heavy?
- Write-heavy?

---

### Mapping

| Workload     | Strategy |
|--------------|----------|
| Read-heavy   | Cache + replicas |
| Write-heavy  | LSM DB (Cassandra) |

---

## 🧠 Insight

> Read vs write decides index strategy + DB type

---

## 4. CONSISTENCY REQUIREMENT

---

### Ask:

- Can system tolerate stale data?

---

### Mapping

| Requirement         | DB Choice |
|---------------------|-----------|
| Strong consistency  | SQL DB    |
| Eventual consistency| Cassandra / DynamoDB |

---

## 🚨 Rule

> Never use eventual consistency for money/inventory

---

## 5. SCALE REQUIREMENT

---

### Ask:

- Data size?
- QPS?

---

### Mapping

| Scale         | Strategy |
|---------------|----------|
| Small         | Single DB |
| Medium        | Replication |
| Large         | Sharding |

---

## 🧠 Insight

> Scale determines architecture, not just DB

---

## 6. SCHEMA FLEXIBILITY

---

### Ask:

- Fixed schema or evolving?

---

### Mapping

| Requirement     | DB |
|-----------------|----|
| Fixed schema    | SQL |
| Flexible schema | MongoDB |

---

---

# ⚖️ IMPORTANT DECISION COMPARISONS

---

## SQL vs NoSQL

---

### Use SQL when:
- Strong consistency needed
- Structured data
- Transactions required

---

### Use NoSQL when:
- Massive scale
- Flexible schema
- High throughput

---

---

## B+ TREE vs LSM TREE

---

### Use B+ Tree when:
- Read-heavy
- Range queries

---

### Use LSM when:
- Write-heavy
- High ingestion

---

---

## CACHE vs DATABASE

---

### Cache (Redis):
- Fast access
- Temporary data

---

### Database:
- Source of truth
- Durable

---

## 🚨 Rule

> Cache is NOT a database

---

---

## NORMALIZATION vs DENORMALIZATION

---

### Normalize when:
- Data integrity critical

---

### Denormalize when:
- Read-heavy systems
- High scale

---

---

## LEADER-FOLLOWER vs LEADERLESS

---

### Leader-Follower:
- Strong consistency
- Simpler

---

### Leaderless:
- High availability
- Eventual consistency

---

---

# 🧠 FINAL DECISION FLOW (INTERVIEW READY)

---

## Step-by-step answer:

1. Define access pattern  
2. Identify workload (OLTP/OLAP)  
3. Check read/write ratio  
4. Decide consistency level  
5. Estimate scale  
6. Choose DB type  
7. Add caching / replication / sharding  

---

## 🗣️ INTERVIEW TEMPLATE

---

“I would first understand access patterns, then evaluate consistency and scale requirements. Based on that, I would choose <DB type>, and then apply replication, caching, and sharding as needed.”

---

---

# 🔥 REAL-WORLD STACK (IMPORTANT)

---

Typical production system:

- Primary DB → PostgreSQL
- Cache → Redis
- Search → Elasticsearch
- Analytics → BigQuery

---

---

# 🚨 COMMON MISTAKES

---

- Choosing DB before access pattern
- Overengineering early
- Using single DB for all use cases
- Ignoring scaling strategy
- Blindly choosing NoSQL

---

---

# 🧠 KEY INSIGHT (MOST IMPORTANT)

---

> Database choice is NOT about features  
> It is about trade-offs under constraints
