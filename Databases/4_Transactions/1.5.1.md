# ⚙️ 1.5.1 CONCURRENCY CONTROL (HOW TRANSACTIONS ACTUALLY WORK)

---

## 📘 Definition (textbook-style)

**Concurrency control** is the mechanism used by databases to ensure that multiple transactions can execute safely and correctly without violating data consistency.

---

## 💡 Intuition (Why it exists)

Multiple users accessing same data:

👉 Example:

* T1 reads balance = 100
* T2 updates balance → 50
* T1 writes based on old value → ❌ incorrect

> Problem: **Concurrent execution causes anomalies**

---

## 🧩 Two Main Approaches

---

# 1. 🔒 PESSIMISTIC CONCURRENCY CONTROL (LOCKING)

---

## 📘 Definition

Transactions acquire locks before accessing data, preventing others from modifying it.

---

## 🔐 Types of Locks

---

### 🔹 Shared Lock (S)

* Read only
* Multiple transactions allowed

---

### 🔹 Exclusive Lock (X)

* Write lock
* Only one transaction allowed

---

## 🔄 Example

```sql
BEGIN;
SELECT * FROM users WHERE id = 1 FOR UPDATE;
```

👉 Locks row → no one else can modify

---

## ⚠️ Problems

---

### 1. Deadlocks

* T1 waits for T2
* T2 waits for T1
  → system stuck

---

### 2. Reduced concurrency

* Too many locks → slow system

---

## 🌍 Real-world Mapping

* MySQL (InnoDB locking)
* PostgreSQL (also supports locking)

---

## ⚖️ Trade-offs

| Pros                 | Cons      |
| -------------------- | --------- |
| Strong consistency   | Blocking  |
| Simple to understand | Deadlocks |

---

# 2. 🔄 OPTIMISTIC CONCURRENCY CONTROL (MVCC)

---

## 📘 Definition

**MVCC (Multi-Version Concurrency Control)** allows multiple versions of data so reads don’t block writes and vice versa.

---

## 💡 Core Idea

> “Instead of locking, create multiple versions”

---

## 🔄 How it works

* Each write creates a **new version**
* Reads see a **snapshot** of data

---

### 🧠 Example

* T1 reads version V1
* T2 updates → creates V2
* T1 still sees V1

👉 No blocking!

---

## ⚡ Benefits

* High concurrency
* No read-write blocking

---

## ⚠️ Problems

* More storage (multiple versions)
* Cleanup needed (vacuum/compaction)

---

## 🌍 Real-world Mapping

* PostgreSQL → MVCC
* MySQL → MVCC (InnoDB)

---

## ⚖️ Locking vs MVCC (VERY IMPORTANT)

| Feature          | Locking (Pessimistic) | MVCC (Optimistic) |
| ---------------- | --------------------- | ----------------- |
| Read/Write block | Yes                   | No                |
| Concurrency      | Lower                 | Higher            |
| Complexity       | Lower                 | Higher            |
| Use case         | High conflict         | Low conflict      |

---

## 🎯 When to use what

---

### Use Locking when:

* High contention (many writes to same data)
* Strict control required

---

### Use MVCC when:

* Read-heavy systems
* High concurrency needed

---

## 🧠 Key Insight (INTERVIEW GOLD)

> “Modern databases use MVCC to allow high concurrency by avoiding read-write locks and serving snapshot-based reads.”

---

## 🗣️ Interview One-liner

> “Concurrency control ensures correctness in concurrent transactions, typically using locking or MVCC, with modern systems favoring MVCC for better scalability.”

---

## 🚨 Red Flags / Mistakes

* ❌ Thinking isolation = concurrency control
* ❌ Ignoring deadlocks
* ❌ Assuming locks are always bad
* ❌ Not knowing MVCC exists
* ❌ Confusing MVCC with eventual consistency

---

## 🔥 Interview Insight (VERY IMPORTANT)

👉 If interviewer asks:

> “How does PostgreSQL handle concurrency?”

Answer:

> “It uses MVCC, allowing snapshot reads without blocking writes.”

---

## ✍️ Quick Mental Model

> Locking → block others
> MVCC → give each transaction its own view
