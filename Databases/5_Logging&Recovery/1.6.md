# ⚙️ 1.6 LOGGING & RECOVERY (WAL + CRASH RECOVERY)

---

## 📘 Definition (textbook-style)

**Logging and recovery** are mechanisms that ensure a database can recover to a consistent state after a crash, using logs that record changes before they are applied to the actual data.

---

## 💡 Intuition (Why it exists)

Disk writes are not instant or atomic:

👉 Scenario:

* Write partially completes
* System crashes
* Data becomes corrupted ❌

> Problem: “How do we ensure durability and recover from failures?”

---

## 🧩 Core Idea

> **Never modify data directly without logging first**

---

## 🔥 Write-Ahead Logging (WAL) — CORE CONCEPT

---

## 📘 Definition

**WAL (Write-Ahead Log)** ensures that all changes are first written to a log before being applied to the database.

---

## 🔄 Write Flow (VERY IMPORTANT)

---

### Step-by-step

1. Transaction makes a change
2. Change is written to **WAL (disk)**
3. WAL is flushed (fsync)
4. Then actual data page is updated

---

### 🧠 Key Rule

> “Log first, data later”

---

## ⚡ Why WAL works

---

Even if crash happens:

* WAL has the record of changes
* DB can recover using logs

---

## 🧩 Types of Logs

---

### 1. Redo Log

---

#### 📘 Purpose

Reapply committed changes after crash

---

#### 💡 Example

* Transaction committed
* Data not written to disk yet

👉 Redo ensures it is applied

---

---

### 2. Undo Log

---

#### 📘 Purpose

Rollback uncommitted changes

---

#### 💡 Example

* Transaction started
* Crash happened before commit

👉 Undo removes partial changes

---

---

## 🔄 Crash Recovery Process

---

### After restart:

---

### 1. Analysis Phase

* Identify committed vs uncommitted transactions

---

### 2. Redo Phase

* Reapply committed changes

---

### 3. Undo Phase

* Rollback incomplete transactions

---

### 🧠 Key Insight

> Database returns to last **consistent state**

---

## ⚖️ Trade-offs

---

| Benefit        | Cost                  |
| -------------- | --------------------- |
| Durability     | Extra disk writes     |
| Crash recovery | Log storage overhead  |
| Reliability    | Write latency (fsync) |

---

## 🌍 Real-world Mapping

---

* PostgreSQL → WAL-based
* MySQL → WAL (InnoDB redo log)
* RocksDB → WAL + LSM

---

## 🎯 When to use vs not use

---

### WAL is used when:

* Need durability (almost always in DBs)

---

### Relax WAL (rare cases):

* High-performance systems
* Acceptable data loss

👉 Example:

* Caching systems → Redis (can be configured)

---

## 🧠 Key Insight (INTERVIEW GOLD)

> “Durability in databases is achieved through write-ahead logging, ensuring that even if data pages are not written, committed transactions can be recovered from logs.”

---

## 🗣️ Interview One-liner

> “WAL ensures durability by logging changes before applying them, enabling recovery through redo and undo operations after a crash.”

---

## 🚨 Red Flags / Mistakes

* ❌ Thinking DB writes directly to disk safely
* ❌ Ignoring WAL fsync cost
* ❌ Not understanding redo vs undo
* ❌ Assuming commit = data written to main storage
* ❌ Ignoring crash recovery phases

---

## 🔥 Advanced Insight (VERY IMPORTANT)

👉 Commit does NOT mean:

* Data is written to main DB ❌

👉 It means:

* Data is safely in WAL ✅

---

## ✍️ Quick Mental Model

> Write → WAL → Commit → Data update later
