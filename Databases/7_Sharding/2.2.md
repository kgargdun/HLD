# 🌐 2.2 SHARDING (PARTITIONING — SCALE OUT)

---

## 📘 Definition (textbook-style)

**Sharding (partitioning)** is the process of splitting a database into smaller, independent pieces (shards) distributed across multiple machines, where each shard holds a subset of the data.

---

## 💡 Intuition (Why it exists)

Single DB node:

* ❌ Storage limit
* ❌ CPU/RAM bottleneck
* ❌ Write throughput limited

👉 Vertical scaling (bigger machine) doesn’t scale infinitely

> Solution: **Scale horizontally by splitting data**

---

## 🧩 Core Idea

> Instead of 1 big DB → N smaller DBs

Each shard:

* Own storage
* Own compute
* Handles part of data

---

## 🔄 How Sharding Works

---

### Example

Users table split across shards:

```
Shard 1 → user_id 1–1M  
Shard 2 → user_id 1M–2M  
Shard 3 → user_id 2M–3M  
```

---

### Query flow

1. Determine shard (using shard key)
2. Route query to correct shard
3. Fetch result

---

## 🧠 Shard Key (VERY IMPORTANT)

---

## 📘 Definition

A **shard key** is the attribute used to decide which shard stores a particular row.

---

## 💡 Examples

* user_id
* region
* timestamp

---

### 🧠 Key Insight

> Shard key determines distribution, performance, and scalability

---

## ⚠️ Bad shard key = system failure

---

## 🧩 Partitioning Strategies

---

# 1. 🔢 HASH-BASED SHARDING

---

## 📘 Definition

Shard = hash(key) % N

---

### 💡 Example

```
hash(user_id) % 4 → shard number
```

---

### ✅ Pros

* Even distribution
* Avoids hotspots

---

### ❌ Cons

* No range queries
* Hard to rebalance

---

---

# 2. 📈 RANGE-BASED SHARDING

---

## 📘 Definition

Data divided based on value ranges

---

### 💡 Example

```
A–F → shard 1  
G–M → shard 2  
N–Z → shard 3  
```

---

### ✅ Pros

* Efficient range queries
* Easy to reason

---

### ❌ Cons

* Hotspots (uneven distribution)

---

---

# 3. 🌍 GEO-BASED SHARDING

---

## 📘 Definition

Partition data by geographic region

---

### 💡 Example

* India → shard 1
* US → shard 2

---

### ✅ Pros

* Low latency
* Data locality

---

### ❌ Cons

* Cross-region queries expensive

---

---

# 4. 🔄 CONSISTENT HASHING (ADVANCED)

---

## 📘 Definition

Distributes data across nodes with minimal reshuffling when nodes change.

---

### 💡 Why needed?

Normal hashing:

* Add/remove node → massive reshuffling ❌

Consistent hashing:

* Only small portion moves ✅

---

## ⚖️ Strategy Comparison

| Strategy   | Pros              | Cons                    |
| ---------- | ----------------- | ----------------------- |
| Hash       | Even distribution | No range queries        |
| Range      | Range queries     | Hotspots                |
| Geo        | Low latency       | Cross-region complexity |
| Consistent | Easy scaling      | More complex            |

---

## ⚠️ Hard Problems in Sharding (VERY IMPORTANT)

---

### 1. Rebalancing

* Adding new shard
* Moving data

👉 Expensive + tricky

---

### 2. Hot Partitions

* Uneven traffic
* One shard overloaded

---

### 3. Cross-shard Queries

* Joins across shards → very expensive

---

### 4. Transactions

* Distributed transactions are complex

---

## 🌍 Real-world Mapping

---

* Cassandra → hash-based partitioning
* DynamoDB → partition key based
* MongoDB → shard key

---

## 🎯 When to use vs not use

---

### Use sharding when:

* Data too large for one machine
* Write throughput too high
* Need horizontal scaling

---

### Avoid sharding when:

* Data fits in one machine
* Complex queries/joins required
* Early-stage system

---

## 🧠 Key Insight (INTERVIEW GOLD)

> “Sharding improves scalability but shifts complexity to application logic, especially around routing, rebalancing, and cross-shard operations.”

---

## 🗣️ Interview One-liner

> “Sharding horizontally partitions data across multiple nodes using a shard key, enabling scalability but introducing challenges like rebalancing and cross-shard queries.”

---

## 🚨 Red Flags / Mistakes

* ❌ Choosing bad shard key
* ❌ Ignoring hotspots
* ❌ Doing cross-shard joins
* ❌ Not planning for rebalancing
* ❌ Sharding too early

---

## 🔥 Interview Insight (VERY IMPORTANT)

👉 Most systems:

* Start unsharded
* Add replication
* Then shard when needed

---

## ✍️ Quick Mental Model

> Sharding = split data
>
> Shard key = routing logic
