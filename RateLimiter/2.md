# Rate Limiting — Algorithms (Interview Core)

> ⚠️ This is the **most important file**.
> Interviews are usually about choosing the right algorithm + explaining tradeoffs.

---

## 1. Fixed Window Counter

### Idea

* Divide time into fixed windows (e.g., 1 minute)
* Maintain a counter per window

### Example

* Limit: 100 req/min
* Window: 10:00–10:01 → 100 requests allowed
* At 10:01 → counter resets

---

### Pros

* Simple to implement
* Low memory
* Fast (O(1))

### Cons

* **Burst problem** ❗

  * 100 requests at 10:00:59
  * 100 requests at 10:01:01
    → 200 requests in ~2 seconds

---

### When to Use

* When precision is not critical
* Basic systems / low traffic

---

## 2. Sliding Window Log

### Idea

* Store timestamp of each request
* Only count requests within last `T` time

---

### Example

* Limit: 100 req/min
* Store timestamps
* Remove entries older than 60 sec

---

### Pros

* Accurate
* No burst issue

### Cons

* High memory (stores every request)
* Expensive cleanup (O(n))

---

### When to Use

* Low traffic systems
* When strict correctness is required

---

## 3. Sliding Window Counter (Optimized)

### Idea

* Approximate sliding window using **two buckets**

  * Current window
  * Previous window (weighted)

---

### Formula

```
effective_count =
  current_count +
  (previous_count * remaining_time_fraction)
```

---

### Pros

* Good balance: accuracy vs memory
* Avoids burst spikes

### Cons

* Slight approximation error

---

### When to Use

* Most real-world systems needing accuracy + performance

---

## 4. Token Bucket ⭐ (MOST IMPORTANT)

### Idea

* Tokens are added at a fixed rate
* Each request consumes 1 token
* Bucket has max capacity

---

### Behavior

* If tokens available → allow
* If empty → reject

---

### Key Feature

* **Allows bursts** (up to bucket size)

---

### Example

* Refill: 10 tokens/sec
* Capacity: 100 tokens

→ Can handle burst of 100 instantly

---

### Pros

* Smooth rate limiting
* Supports bursts naturally
* Widely used in production

### Cons

* Slightly more complex than counters

---

### When to Use

* APIs with burst traffic
* User-facing systems

---

## 5. Leaky Bucket

### Idea

* Requests enter a queue
* Processed at constant rate

---

### Behavior

* Incoming rate can be bursty
* Outgoing rate is smooth

---

### Pros

* Smooth traffic
* Good for downstream protection

### Cons

* Can introduce latency (queueing)
* Drops requests if queue full

---

### When to Use

* Traffic shaping
* Protecting backend services

---

## 6. Token Bucket vs Leaky Bucket

| Feature       | Token Bucket | Leaky Bucket        |
| ------------- | ------------ | ------------------- |
| Burst support | ✅ Yes        | ❌ No (smoothed out) |
| Output rate   | Variable     | Constant            |
| Use case      | APIs         | Network shaping     |

---

## 7. What to Say in Interviews (IMPORTANT)

### If interviewer asks:

👉 “Design a rate limiter”

Start with:

* Simple → Fixed Window
* Improve → Sliding Window
* Production → **Token Bucket**

---

### Golden Answer Pattern

> “I would use a Token Bucket algorithm backed by Redis. It allows controlled bursts while maintaining a steady rate, and is efficient for distributed systems.”

---

## 8. Complexity Summary

| Algorithm       | Time | Space | Accuracy |
| --------------- | ---- | ----- | -------- |
| Fixed Window    | O(1) | Low   | Low      |
| Sliding Log     | O(n) | High  | High     |
| Sliding Counter | O(1) | Low   | Medium   |
| Token Bucket    | O(1) | Low   | High     |
| Leaky Bucket    | O(1) | Low   | High     |

---

## 9. Common Follow-ups

* How to implement Token Bucket in distributed system?
* What if Redis fails?
* How to handle clock drift?
* What about multi-region limits?

---

## 10. One-Line Summary

> Fixed Window is simplest, Sliding Window improves accuracy, but Token Bucket is the industry standard due to burst handling + efficiency.
