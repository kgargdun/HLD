# Rate Limiting — Distributed Design (Interview Focus)

> This is where most candidates struggle.
> Single-node rate limiting is easy — **distributed makes it real HLD**.

---

## 1. Problem Statement

In distributed systems:

* Multiple API servers handle requests
* Each server sees only **partial traffic**

👉 Challenge:

> How do we enforce a **global rate limit** consistently?

---

## 2. Naive Approach (WRONG)

Each server maintains its own counter:

```
User limit = 100 req/min
Server A sees 60
Server B sees 60
→ Total = 120 ❌ (limit violated)
```

---

## 3. Centralized Store (Baseline Solution)

Use a **shared datastore** (most commonly Redis)

### Flow

1. All servers hit Redis
2. Increment counter
3. Check limit
4. Allow / reject

---

### Why Redis?

* In-memory → fast
* Atomic ops (INCR, LUA scripts)
* TTL support

---

## 4. Key Design Decisions

### (a) Key Design

```
rate_limit:{userId}:{window}
```

Examples:

* `rate_limit:123:10:00`
* `rate_limit:IP_1.2.3.4`

---

### (b) Sharding

* Distribute keys across Redis cluster
* Hash(userId/IP)

👉 Avoids single-node bottleneck

---

## 5. Atomicity Problem

### Issue

Two requests at same time:

* Both read count = 99
* Both increment → 101 ❌

---

### Solution

Use atomic operations:

* Redis `INCR`
* Or Lua scripts

👉 Ensures correctness

---

## 6. Consistency vs Availability Tradeoff

### Strong Consistency

* Single central store
* Accurate limits
* Higher latency

---

### Eventual Consistency

* Local counters + sync
* Faster
* May allow slight overuse

---

### Interview Insight

> “We can allow slight limit violations for better availability and latency.”

---

## 7. Global vs Regional Rate Limiting

### Regional (Common)

* Each region enforces limits independently
* Faster

### Global (Hard)

* Requires cross-region coordination
* High latency

---

### Practical Approach

* Use **regional limits**
* Sync periodically

---

## 8. Clock Synchronization Problem

### Issue

* Different servers have different clocks
* Window boundaries mismatch

---

### Solutions

* Use:

  * Redis server time
  * Or epoch-based windows

---

## 9. Hot Key Problem 🔥

### Issue

* Popular users/IPs → single key becomes hotspot

---

### Solutions

#### (a) Key Sharding

```
rate_limit:{userId}:{shardId}
```

#### (b) Randomized Keys

* Spread load across multiple counters

#### (c) Local + Global Hybrid

* Local pre-check before Redis hit

---

## 10. Failure Handling

### Case: Redis Down

Options:

1. **Fail Open** (allow requests)

   * Better UX
   * Risk abuse

2. **Fail Closed** (block requests)

   * Safe
   * Bad UX

---

### Interview Answer

> “Depends on system criticality. For login → fail closed, for general APIs → fail open.”

---

## 11. Optimization: Reduce Redis Calls

### Problem

Every request → Redis call → expensive

---

### Solutions

#### (a) Local Cache

* Small in-memory counters

#### (b) Token Prefetching

* Fetch tokens in batch from Redis

---

## 12. Typical Production Architecture

```
Client
  ↓
API Gateway
  ↓
Rate Limiter Service
  ↓
Redis Cluster
```

---

## 13. What to Say in Interviews

> “I would use a centralized Redis-based rate limiter with atomic operations. For scale, I’d shard keys, handle hot keys via partitioning, and optionally use local caching to reduce Redis load.”

---

## 14. Common Follow-ups

* How do you scale Redis?
* What if one region exceeds limit?
* How to do multi-tenant rate limiting?
* Can we make it eventually consistent?

---

## 15. One-Line Summary

> Distributed rate limiting relies on a shared fast store (like Redis), with careful handling of atomicity, sharding, and tradeoffs between consistency and performance.
