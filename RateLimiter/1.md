# Rate Limiting — Basics (Interview Focus)

## 1. What is Rate Limiting?

Rate limiting controls **how many requests a client can make in a given time window**.

👉 Example:

* 100 requests per minute per user
* 1000 requests per hour per API key

If limit is exceeded → requests are:

* Rejected (HTTP 429)
* Delayed (queued/throttled)

---

## 2. Why Do We Need It?

### (a) Prevent Abuse

* Bots, brute force, scraping
* DDoS mitigation (first line of defense)

### (b) Ensure Fair Usage

* Prevent few users from hogging resources
* Multi-tenant systems

### (c) Cost Control

* Protect expensive downstream services (DB, ML models, third-party APIs)

### (d) Stability

* Avoid cascading failures under load

---

## 3. Where is it Applied?

### Levels of Rate Limiting

1. **Per User**

   * userId-based
   * e.g., login attempts

2. **Per IP**

   * useful for anonymous traffic

3. **Per API Key**

   * SaaS platforms

4. **Global**

   * protect entire system

5. **Per Resource**

   * e.g., `/login` stricter than `/home`

---

## 4. Hard vs Soft Limits

### Hard Limit

* Strict cutoff
* Example: 100 req/min → 101st request rejected

### Soft Limit

* Allows burst or grace
* Example:

  * 100 req/min allowed
  * small temporary overflow tolerated

👉 Real systems often use **soft limits** for better UX

---

## 5. Key Concepts

### (a) Time Window

* Defines duration (per second/min/hour)

### (b) Identity (Key)

* What are we limiting on?

  * userId / IP / token

### (c) Counter

* Tracks number of requests

---

## 6. Basic Flow

1. Request comes in
2. Extract key (user/IP/API key)
3. Check current count in window
4. If count < limit → allow
5. Else → reject or throttle

---

## 7. Placement in System

### (a) Client Side

* Rare, not reliable

### (b) API Gateway (MOST COMMON)

* Centralized enforcement

### (c) Service Level

* Fine-grained control

---

## 8. Important Interview Notes

* Always clarify:

  * **What are we limiting? (user/IP/API key)**
  * **What time window?**
  * **Strict or burst allowed?**

* Rate limiter is often:

  * **Stateless API layer + state in fast store (Redis)**

* Tradeoff:

  * Accuracy vs performance

---

## 9. Common Pitfalls

* Fixed windows causing burst spikes
* Distributed systems → inconsistent counts
* Clock synchronization issues
* Hot keys (popular users/IPs)

---

## 10. One-Line Summary (for interviews)

> Rate limiting is a mechanism to control request rate per identity over time to ensure fairness, prevent abuse, and maintain system stability.
