## 1. Redis — The Backbone of Most Rate Limiters

### How it's used

* Counter-based limiting:

```bash
INCR key
EXPIRE key 60
```

* Token bucket via Lua scripts

---

### Why Redis dominates

* Atomic operations
* Extremely fast (in-memory)
* TTL support
* Distributed via clustering

---

### Interview Line

> “Most production systems use Redis as the central store for rate limiting.”

---

## 2. NGINX — Leaky Bucket in Action

### Feature

* `limit_req` module

---

### Algorithm

* **Leaky Bucket**

---

### Behavior

* Smooths incoming traffic
* Excess requests:

  * Delayed OR dropped

---

### Use Case

* Protect backend servers

---

### Interview Insight

> “NGINX uses a leaky bucket approach to smooth traffic at the edge.”

## 3. Envoy — Modern Service Mesh Limiting

### Features

* Local + global rate limiting
* Works with sidecars

---

### Architecture

```id="v4yq2k"
Service → Envoy Sidecar → Rate Limit Service → Redis
```

---

### Key Idea

* Decoupled rate limiter service

---

### Why important

* Microservices world
* Fine-grained control

---

## 4. AWS API Gateway — Token Bucket at Scale

### Features

* Request rate limits
* Burst capacity

---

### Algorithm

* **Token Bucket**

---

### Config Example

* Rate: 1000 req/sec
* Burst: 2000

---

### Insight

> “Cloud providers expose token bucket semantics — steady rate + burst capacity.”

---

## 5. Cloudflare — Edge Rate Limiting

### Where it runs

* At CDN edge (closest to user)

---

### Why powerful

* Blocks traffic before it hits origin

---

### Features

* IP-based limiting
* Bot detection
* Global distribution

---

### Insight

> “Edge rate limiting reduces load on backend systems significantly.”

---

## 6. Real Production Pattern (Very Important)

### Multi-Layer Rate Limiting

```id="r3m9kp"
Client
  ↓
CDN (Cloudflare)         ← coarse filtering
  ↓
API Gateway              ← global limits
  ↓
Service-level limiter    ← fine-grained control
  ↓
Redis                    ← shared state
```

---

### Why multiple layers?

* Defense in depth
* Reduce load early
* Different granularity

---

## 7. Real System Examples

### Login API

* Strict limits
* Fail closed
* Per IP + per user

---

### Public API (like Twitter)

* Token bucket
* Per API key
* Burst allowed

---

### Internal Services

* Usually softer limits
* Focus on stability
