# Rate Limiting — Storage & Scaling (Interview Focus)

> This file answers:
> **Where do we store counters and how do we scale the system?**

---

## 1. Storage Options

### (a) In-Memory Store (Primary Choice)

Most common: Redis

---

### Why In-Memory?

* Ultra low latency (sub-ms)
* High throughput
* Built-in primitives:

  * `INCR`
  * `EXPIRE`
  * Lua scripting

---

### Example

```id="xq91ld"
INCR rate_limit:user123
EXPIRE rate_limit:user123 60
```

---

### Pros

* Fast
* Simple
* Scales horizontally

### Cons

* Memory cost
* Data loss (acceptable for rate limiting)

---

## 2. Persistent Storage (Rare)

Examples:

* SQL / NoSQL DBs

---

### Why NOT preferred?

* High latency
* Cannot handle high QPS
* Overkill for ephemeral data

---

### When Used?

* Audit logs
* Analytics (not enforcement path)

---

## 3. TTL (Time-To-Live) — CRITICAL

### Idea

* Automatically delete counters after window expires

---

### Why Needed?

* Prevent memory bloat
* Avoid manual cleanup

---

### Example

```id="4ij0xt"
SET key 1 EX 60
```

---

## 4. Scaling the Rate Limiter

### (a) Horizontal Scaling

* Multiple API servers
* Shared Redis cluster

---

### (b) Redis Cluster

* Partition keys across nodes
* Hash-based sharding

---

### Key Insight

> Rate limiter scalability = Redis scalability

---

## 5. Hot Key Problem 🔥 (Deep Dive)

### Problem

* Some users/IPs receive massive traffic
* Single key → single shard → bottleneck

---

### Solutions

#### (a) Key Splitting

```id="2n8f9v"
rate_limit:user123:0
rate_limit:user123:1
rate_limit:user123:2
```

* Aggregate counts

---

#### (b) Probabilistic Drop

* Randomly reject requests before hitting store

---

#### (c) Local Rate Limiting Layer

* Pre-filter before Redis

---

## 6. Approximation Techniques (Advanced)

### (a) Sliding Window Counter

* Already reduces memory

---

### (b) Probabilistic Structures

* Bloom Filters (rare here)
* Count-Min Sketch

---

### Tradeoff

* Accuracy ↓
* Memory ↓
* Performance ↑

---

### Interview Note

> Mention only if interviewer pushes for “extreme scale”

---

## 7. Reducing Load on Redis

### Problem

Every request → network call

---

### Solutions

#### (a) Local Cache

* Maintain small counters locally
* Sync periodically

---

#### (b) Token Prefetching

* Fetch tokens in batch
* Consume locally

---

#### (c) Batching

* Combine multiple updates

---

## 8. Multi-Level Rate Limiting

### Pattern

```id="g6d8qp"
Local Limiter (fast, approximate)
        ↓
Global Limiter (accurate, Redis)
```

---

### Benefits

* Reduced load on Redis
* Faster response

---

## 9. Memory Management

### Issues

* Millions of keys
* Short-lived data

---

### Solutions

* TTL-based eviction
* LRU eviction (Redis config)
* Key compression (short keys)

---

## 10. Throughput Considerations

### Bottlenecks

* Network latency
* Redis CPU
* Hot shards

---

### Optimizations

* Pipeline requests
* Use connection pooling
* Co-locate services with Redis

---

## 11. What to Say in Interviews

> “I would use Redis with TTL-based keys for counters. For scaling, I’d use Redis cluster with sharding, handle hot keys via partitioning, and reduce load using local caching or token prefetching.”

---

## 12. Common Follow-ups

* How do you handle Redis memory limits?
* What eviction strategy to use?
* How to reduce network calls?
* Can we avoid Redis entirely?

---

## 13. One-Line Summary

> Rate limiter storage is typically in-memory (Redis) with TTL-based counters, and scaling is achieved via sharding, caching, and multi-level limiting.
