# 📄 **File 4 — Quorum & Tunable Consistency**

---

## 🧠 1. Why Quorum Exists

From File 3:

👉 In **leaderless systems**, any node can accept writes
👉 Now problem = **how many replicas should agree?**

---

### Core Question:

> “How many nodes must respond before we consider an operation successful?”

👉 That answer = **Quorum**

---

## ⚖️ 2. What is Quorum?

> A quorum is the **minimum number of nodes that must agree** for a read/write to succeed.

---

### Basic Setup

* Total replicas = **N**
* Write quorum = **W**
* Read quorum = **R**

---

## 🧩 3. The Golden Rule (MUST MEMORIZE)

👉 **R + W > N**

---

### Why?

Ensures:

> Read and write sets **overlap**

So:

* At least one node has latest data

---

## Example

```
N = 3
W = 2
R = 2
```

👉 R + W = 4 > 3 ✅

So:

* Reads will see latest write

---

## ❌ If Not:

```
N = 3
W = 1
R = 1
```

👉 R + W = 2 ≤ 3 ❌

→ Reads may miss latest write

---

## 🎯 4. What Quorum Gives You

---

### ✔ Strong Consistency (if configured correctly)

When:

* R + W > N

---

### ✔ Tunable Tradeoffs

You can adjust:

* latency
* consistency
* availability

---

# ⚙️ 5. Tunable Consistency (INTERVIEW GOLD)

---

## Case 1: Write-heavy system

```
W = 1
R = N
```

✔ Fast writes
❌ Slow reads
❌ Risk of stale reads

---

## Case 2: Read-heavy system

```
W = N
R = 1
```

✔ Fast reads
❌ Slow writes

---

## Case 3: Balanced

```
W = 2
R = 2 (for N=3)
```

✔ Strong consistency
✔ Moderate latency

---

## Real Systems

* Apache Cassandra
* Amazon DynamoDB

---

## Interview Line

> “We tune R and W based on workload to balance latency and consistency.”

---

# ⏳ 6. What Happens During Writes

---

### Example (N=3, W=2):

1. Client sends write to 3 nodes
2. 2 nodes acknowledge
3. Write is considered successful

---

👉 Third node:

* May update later (eventual consistency)

---

# 🔍 7. What Happens During Reads

---

### Example (R=2):

1. Read from 2 nodes
2. Compare versions
3. Return latest

---

👉 If mismatch:

* trigger **read repair**

---

# ⚠️ 8. Failure Scenarios

---

## Case 1: Node down

```
N = 3
W = 2
```

1 node fails → still works ✅

---

## Case 2: Too many failures

2 nodes fail:

❌ Cannot reach quorum
→ system unavailable

---

## Insight

> Quorum trades availability for consistency

---

# 💥 9. Split Brain Problem

---

## What is it?

Network partition divides cluster:

```
Group A (2 nodes)
Group B (1 node)
```

---

## Problem

Both sides may accept writes → conflicting data

---

## Solutions

* Majority quorum (only majority writes allowed)
* Consensus algorithms (Raft, Paxos)

---

## Real Systems

* Apache Cassandra → quorum-based
* ZooKeeper → strict quorum

---

## Interview Line

> “We avoid split brain by requiring majority quorum for writes.”

---

# 🔄 10. Quorum + Eventual Consistency

---

Even with quorum:

👉 Some replicas may still be stale

So systems use:

* Read repair
* Anti-entropy
* Versioning

---

## Important Insight

> Quorum reduces inconsistency window, but does not eliminate it completely

---

# 🧠 11. Mental Model

---

### Think like this:

* Write → reaches W nodes
* Read → checks R nodes
* Overlap → ensures correctness

---

👉 No overlap = danger

---

# 🎯 12. How to Answer in Interviews

---

## Question:

“How do you ensure consistency in Dynamo-like systems?”

---

### Answer:

1. Use replication (N nodes)

2. Use quorum:

   > “We ensure R + W > N”

3. Handle conflicts:

   > “Using versioning / vector clocks”

4. Tradeoff:

   > “Higher quorum increases latency but improves consistency”

---

# 🔥 13. Common Mistakes

---

### ❌ “Quorum guarantees strong consistency always”

✔ Truth:

> Only if R + W > N and no concurrent writes

---

### ❌ “W = 1 is fine always”

✔ Truth:

> Risk of data loss + stale reads

---

### ❌ “More replicas = better consistency”

✔ Truth:

> More replicas → harder coordination

---

# 🔚 Final Summary

* Quorum = minimum nodes required
* Parameters:

  * N, R, W
* Golden rule:

  * **R + W > N**
* Enables:

  * tunable consistency
* Tradeoffs:

  * latency vs availability vs consistency
* Problems:

  * split brain
  * stale reads
* Fixes:

  * read repair
  * versioning
