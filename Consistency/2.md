# 📄 **File 2 — Consistency Models (Core Interview Section)**

---

## 🧠 1. What is a Consistency Model?

A **consistency model defines rules** for:

* **When writes become visible**
* **In what order operations are seen**

👉 Interview definition:

> “A consistency model specifies guarantees about the visibility and ordering of updates across a distributed system.”

---

## 🔑 2. The Big Spectrum (Memorize This Ladder)

From strongest → weakest:

1. Strong Consistency (Linearizability)
2. Sequential Consistency
3. Causal Consistency
4. Eventual Consistency

👉 Think:

> As you go down → **faster, more available, less strict**

---

# 🥇 3. Strong Consistency (Linearizability)

---

## Definition

> Every read sees the **latest completed write**

System behaves like:
👉 “Only ONE copy of data exists”

---

## Example

```
T1: write(x = 10)
T2: read(x) → MUST return 10
```

No matter:

* which server
* which region

---

## Real Systems

* Google Spanner
* CockroachDB
* PostgreSQL (single node)

---

## When to Use

* Payments
* Banking
* Inventory (last item problem)

---

## Tradeoffs

* ❌ High latency
* ❌ Lower availability during failures
* ✔ No anomalies

---

## Interview Line

> “We ensure linearizability by routing writes through a leader and confirming replication before acknowledging.”

---

# 🥈 4. Sequential Consistency

---

## Definition

> All operations appear in the **same order for everyone**, but not necessarily real-time order

---

## Key Difference from Strong

* Strong → respects **real time**
* Sequential → only **same order**

---

## Example

Possible order:

```
write(x=1)
write(x=2)
```

Everyone sees:

```
1 → 2
```

But timing may differ.

---

## When It Appears

* Multi-threaded systems
* Some distributed DB internals

---

## Tradeoffs

* ✔ Easier than strong consistency
* ❌ Still not very scalable

---

## Interview Insight

> Rarely explicitly chosen, but useful to explain **ordering vs real-time guarantees**

---

# 🥉 5. Causal Consistency (VERY IMPORTANT)

---

## Definition

> If one operation **causes another**, all nodes must see them in that order

---

## Example

User flow:

```
User posts: "Hello"
User edits: "Hello World"
```

👉 Everyone must see:

```
post → edit
```

---

But unrelated operations can vary:

```
User A posts
User B posts
```

👉 Order can differ

---

## Real Systems

* DynamoDB (partial support via design)
* Riak
* Messaging systems

---

## How It’s Implemented

👉 Using:

* Vector Clock

---

## When to Use

* Social media
* Messaging systems
* Collaborative apps

---

## Tradeoffs

* ✔ Preserves logical order
* ✔ Better performance than strong
* ❌ Complex to implement

---

## Interview Line

> “We ensure causality using vector clocks to track dependencies between operations.”

---

# ⚖️ 6. Eventual Consistency

---

## Definition

> If no new updates happen, all replicas will **eventually converge**

---

## Example

* Like count = 100
* Some users see 98, 102 temporarily
* Eventually → everyone sees 100

---

## Real Systems

* Apache Cassandra
* Amazon DynamoDB
* Facebook (feeds, likes)

---

## When to Use

* Feeds
* Analytics
* Recommendations

---

## Tradeoffs

* ✔ Very fast
* ✔ Highly available
* ❌ Temporary inconsistency
* ❌ Conflict resolution needed

---

## Interview Line

> “We accept temporary inconsistency and resolve conflicts using LWW or versioning.”

---

# 🔄 7. Practical Session-Level Guarantees (IMPORTANT)

These are **used in real systems a lot**

---

## 7.1 Read-After-Write Consistency

> You always see your own writes

👉 Example:

* Update profile → immediately visible to you

---

## 7.2 Monotonic Reads

> Once you see a value, you never go backwards

---

## 7.3 Monotonic Writes

> Writes are applied in order

---

## 7.4 Session Consistency

Combines:

* read-after-write
* monotonic reads

---

## Real Systems

* Amazon cart
* Google Docs

---

# 🧠 8. Quick Comparison Table

| Model      | Order Guarantee | Real-time? | Use Case |
| ---------- | --------------- | ---------- | -------- |
| Strong     | Total           | Yes        | Payments |
| Sequential | Total           | No         | Rare     |
| Causal     | Partial         | No         | Social   |
| Eventual   | None            | No         | Feeds    |

# 🎯 9. How to Answer in Interviews

---

## Question:

“What consistency model would you choose?”

---

### Structure your answer:

1. **Requirement**

   > “Users must see latest balance”

2. **Model**

   > “We need strong consistency”

3. **Tradeoff**

   > “Higher latency, but correctness critical”

---

### Another Example:

> “Feed system can tolerate staleness → eventual consistency improves availability and latency”

---

# 🔥 10. Common Interview Traps

---

### ❌ Trap 1:

“Eventual consistency means random behavior”

✔ Fix:

> “It guarantees convergence”

---

### ❌ Trap 2:

“Causal = strong”

✔ Fix:

> “Causal only preserves dependency order”

---

### ❌ Trap 3:

“Strong consistency is always best”

✔ Fix:

> “It reduces availability and increases latency”

---

# 🔚 Final Summary

* Consistency models define:

  * visibility
  * ordering
* Spectrum:

  * strong → eventual
* Most used:

  * strong (critical data)
  * causal (user interactions)
  * eventual (scalable systems)
* Session guarantees are **practical optimizations**
