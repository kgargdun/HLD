# 📄 **File 1 — Fundamentals & Mental Models of Consistency**

---

## 🧠 1. What is Consistency (Real Meaning)

**Consistency = “All users see a logically correct view of data”**

But in distributed systems, that splits into:

* **Same value**
* **Correct order**
* **No contradictions**

👉 Interview version:

> “Consistency defines what guarantees a system provides about visibility and ordering of updates.”

---

## ⚠️ 2. Why Consistency is Hard (Core Insight)

In a **single machine**:

* One memory
* One clock
* One execution order

👉 Life is easy.

In a **distributed system**:

* Multiple machines
* Network delays
* No global clock
* Messages can be:

  * delayed
  * dropped
  * reordered

👉 This creates **uncertainty**

---

### Example:

User updates profile:

* Request hits Server A
* Server B still has old data

Now:

* One user sees **new name**
* Another sees **old name**

👉 System is **inconsistent**

---

## 🌍 3. What Problem Are We Actually Solving?

Consistency is about answering:

> “When multiple copies of data exist, what do users see and when?”

---

### Real-world mapping:

* Instagram
  → Likes count may lag → eventual consistency

* Amazon
  → Cart must be correct → strong consistency

* WhatsApp
  → Message order matters → causal consistency

---

## 🔥 4. Types of Anomalies (VERY IMPORTANT)

These are **interview gold**—use them to justify design decisions.

---

### 4.1 Dirty Read

You read data that is not committed yet.

👉 Example:

* User updates balance → not saved yet
* Another user reads it

❌ Wrong value seen

---

### 4.2 Non-Repeatable Read

Same query → different result

👉 Example:

* Read profile name = “A”
* Another update happens
* Read again → “B”

---

### 4.3 Phantom Read

New rows appear/disappear

👉 Example:

* Query: “all orders > ₹1000”
* Another order added
* Query result changes

---

### 4.4 Lost Update (VERY IMPORTANT)

👉 Example:

* Two users update same field
* One overwrites the other

❌ Data loss

---

## ⚖️ 5. Tradeoffs You MUST Understand

You **cannot have everything**:

| Property     | Meaning              |
| ------------ | -------------------- |
| Consistency  | Same data everywhere |
| Availability | Always responds      |
| Latency      | Fast response        |

---

### Key Insight:

> Improving consistency often increases latency or reduces availability

---

## ⚡ 6. CAP Theorem (Foundation)

### CAP Theorem

States that in distributed systems, you can only choose:

* **C** → Consistency
* **A** → Availability
* **P** → Partition tolerance (mandatory)

👉 So real tradeoff is:

* **CP systems**
* **AP systems**

---

### 6.1 What is Partition?

Network failure between nodes.

👉 Example:

* Node A cannot talk to Node B

Now system must choose:

* Return stale data (AP)
* Reject request (CP)

---

### Real Systems:

* Apache Cassandra → AP (eventual consistency)
* Google Spanner → CP (strong consistency)
* Amazon DynamoDB → tunable (AP/CP hybrid)

---

## 🧩 7. Strong vs Eventual Consistency (First Mental Model)

---

### Strong Consistency

> “Every read gets the latest write”

✔ No confusion
❌ Higher latency

👉 Example:

* Bank balance

---

### Eventual Consistency

> “System will become consistent eventually”

✔ Fast
✔ Highly available
❌ Temporary inconsistency

👉 Example:

* Social media likes

---

## 🧠 8. Key Mental Models (Memorize These)

---

### 🧩 Model 1: “Time Gap Problem”

* Write happens now
* Other nodes learn later

👉 Consistency defines:
**What happens during this gap**

---

### 🧩 Model 2: “Multiple Truths”

* Each replica may have different data

👉 Consistency decides:
**Which truth is acceptable**

---

### 🧩 Model 3: “User Experience”

Think in terms of:

* Can user see stale data?
* Can user see wrong order?
* Can user lose data?

---

## 🎯 9. How to Use This in Interview

When asked:

👉 “What consistency do you want?”

Answer like this:

1. Define requirement:

   * “Users should always see latest balance”

2. Translate:

   * “We need strong consistency”

3. Tradeoff:

   * “This increases latency, but correctness is critical”

---

## 🔚 Final Summary (Revision)

* Consistency = guarantees about **visibility + ordering**
* Distributed systems = **no global truth**
* Problems:

  * stale reads
  * lost updates
* Tradeoff:

  * consistency vs availability vs latency
* CAP = you can’t have all 3
* Strong vs Eventual = first big decision
