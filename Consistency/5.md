# 📄 **File 5 — Conflict Resolution & Ordering**

---

## 🧠 1. Why Conflicts Happen

Conflicts occur when:

* Multiple replicas accept writes
* At the **same time**
* Without coordination

---

### Example (Classic)

User updates profile from two devices:

```
Device A → name = "Kaustubh A"
Device B → name = "Kaustubh B"
```

Both reach different replicas.

👉 Now system has **two truths**

---

## ⚠️ Core Problem

> “Which write is correct?”

---

## 🔥 2. Types of Conflicts

---

### 2.1 Concurrent Writes (MOST IMPORTANT)

* No ordering between operations
* Happen in:

  * leaderless systems
  * multi-leader systems

---

### 2.2 Out-of-order Writes

* Network delay changes arrival order

---

### 2.3 Duplicate Writes

* Retries from client

---

---

# ⚖️ 3. Conflict Resolution Strategies

---

## 🥇 3.1 Last Write Wins (LWW)

---

### Idea

> The write with the **latest timestamp wins**

---

### Example

```
T1: name = "A"
T2: name = "B"
```

👉 Keep "B"

---

### Used in

* Apache Cassandra
* Amazon DynamoDB

---

### Problem

👉 Requires **time**

But in distributed systems:

> No perfectly synchronized clock

---

### Risk

❌ Data loss
❌ Wrong ordering

---

## Interview Line

> “LWW is simple but can overwrite valid concurrent updates.”

---

---

## 🥈 3.2 Versioning (Multi-Version)

---

### Idea

Store multiple versions instead of overwriting

---

### Example

```
name = ["A", "B"]
```

👉 System returns both → application resolves

---

### Used in

* Riak

---

### Tradeoffs

✔ No data loss
❌ More complexity
❌ App must resolve conflicts

---

---

## 🥉 3.3 Application-Level Resolution

---

### Idea

Let business logic decide

---

### Examples

* Shopping cart → merge items
* Social media → append posts
* Counters → increment safely

---

### Used in

* Amazon cart
* Facebook feeds

---

## Interview Line

> “We push conflict resolution to application logic for domain-specific correctness.”

---

---

# 🧠 4. Ordering Problem (THE REAL ISSUE)

---

👉 Conflict resolution depends on:

> “Which write happened before the other?”

---

But in distributed systems:

* No global clock ❌
* Events happen in parallel ❌

---

👉 So we need:
**Logical ordering**

---

---

# 🔑 5. Vector Clocks (CRITICAL TOPIC)

---

### Vector Clock

---

## 🧩 Idea

Track:

> “What events has this node seen?”

---

### Structure

Each node maintains:

```
{NodeA: 2, NodeB: 1}
```

---

Meaning:

* Seen 2 updates from A
* Seen 1 update from B

---

---

## ⚙️ How It Works (Step-by-Step)

---

### Step 1: Initial

```
A: {A:0}
B: {B:0}
```

---

### Step 2: A writes

```
A: {A:1}
```

---

### Step 3: B writes

```
B: {B:1}
```

---

### Step 4: Compare

Now we have:

```
A: {A:1}
B: {B:1}
```

👉 Neither dominates → **concurrent**

---

## 💥 Key Insight

> If neither vector is greater → conflict

---

---

## 🧠 Comparison Rule

---

### Case 1: One dominates

```
{A:2, B:1} > {A:1, B:1}
```

✔ Ordered
→ newer version

---

### Case 2: Concurrent

```
{A:1} vs {B:1}
```

❌ No ordering
→ conflict

---

---

## 🎯 What Vector Clocks Solve

---

✔ Detect concurrent updates
✔ Avoid wrong overwrites
✔ Enable safe conflict resolution

---

## ❌ What They Don’t Solve

---

❌ They don’t resolve conflicts automatically
❌ They only **detect**

---

---

* Amazon Dynamo

# 🔄 6. Putting It All Together

---

### Flow in real system:

1. Write happens on multiple nodes
2. Each version tagged with vector clock
3. On read:

   * Compare versions
   * Detect conflict
4. Resolve using:

   * LWW OR
   * merge OR
   * app logic

---

---

# 🧠 7. Mental Model

---

👉 Think:

* Replication → creates multiple versions
* No clock → cannot rely on time
* Vector clock → tells relationship
* Resolution → business decision

---

---

# 🎯 8. How to Answer in Interviews

---

## Question:

“How do you handle concurrent updates?”

---

### Answer:

1. Detect conflict:

   > “Using vector clocks”

2. Resolve:

   > “Using LWW or application-level merging”

3. Tradeoff:

   > “LWW is simple but risks data loss, so critical systems use custom resolution”

# 🔥 9. Common Mistakes

---

### ❌ “Vector clocks give ordering”

✔ Truth:

> They detect ordering or concurrency

---

### ❌ “LWW is always fine”

✔ Truth:

> Causes silent data loss

---

### ❌ “We can use timestamps reliably”

✔ Truth:

> Clocks are not perfectly synced

---

---

  * versioning
  * application logic
* Core problem = ordering
* Vector clocks:

  * detect concurrency
  * enable safe resolution
* Real systems combine:

  * quorum + vector clocks + resolution
