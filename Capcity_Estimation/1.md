# Capacity Estimation (HLD)

---

# 0. Definitions (no assumptions)

* **DAU (Daily Active Users)**: users active per day
* **QPS (Queries Per Second)**: requests handled per second
* **RPS (Requests Per Second)**: same as QPS (used interchangeably)
* **BOTEC (Back-of-the-envelope calculation)**: quick approximation, not exact 

Goal:

> Estimate scale fast → design system accordingly

---

# 1. End-to-End Flow

## 1.1 Users → Requests

### Theory

System load originates from users.

Each user performs actions → actions become requests.

### Formula

* Requests/day = DAU × actions per user
* Avg QPS = requests / 86400
* Peak QPS = avg × (3 to 5)

### Why peak?

Traffic is uneven (evening spikes, events).

### Memory hook

> Users → actions → peak load

---

## 1.2 Requests → Resource Cost

### Theory

Each request consumes:

* CPU (compute)
* Memory (RAM)
* IO (disk/network)

Types:

1. **CPU-bound** → computation heavy
2. **Memory-bound** → RAM access
3. **IO-bound** → disk/network (slowest)

Key relation:

* CPU : Memory : IO ≈ 1 : 10 : 100 

### Example

* Compress 1KB → ~3 microseconds (CPU)
* Read 1MB RAM → ~9 microseconds
* Read 1MB SSD → ~200 microseconds 

### Memory hook

> CPU fast, IO slow, memory in between

---

## 1.3 Requests → Throughput (server capacity)

### Theory

Each server can handle limited QPS.

Typical:

* SQL DB (MySQL): ~1K QPS
* Key-value store: ~10K QPS
* Cache (Redis): 100K – 1M QPS 

### CPU-based estimation (important)

Given:

* 1 request ≈ 1 ms CPU
  → 1 core handles ~1000 requests/sec

* 64 core server → ~64K QPS 

### Memory hook

> 1 core ≈ 1K QPS

---

## 1.4 Requests → Storage

### Theory

Only **writes generate storage**.

### Formula

* Storage/sec = write_QPS × size_per_write
* Total storage = storage/sec × retention

Add:

* replication (×3 usually)

### Units

* 1 KB = 10³ bytes
* 1 MB = 10⁶ bytes
* 1 GB = 10⁹ bytes
* 1 TB = 10¹² bytes
* 1 EB = 10¹⁸ bytes 

### Memory hook

> Writes accumulate forever

---

## 1.5 Requests → Bandwidth

### Theory

Network load = data transferred per second

### Formula

* Bandwidth = QPS × (request + response size)

### Example

* 10K QPS × 10 KB → 100 MB/sec

### Memory hook

> QPS × size = network cost

---

## 1.6 Fanout (Amplification)

### Theory

One write can trigger many reads.

Example:

* Post → 100 followers → 100 reads

### Impact

Read QPS >> Write QPS

### Memory hook

> 1 write = N reads

---

## 1.7 Infra Mapping

### Based on bottleneck:

| Problem         | Solution           |
| --------------- | ------------------ |
| High read load  | Cache / CDN        |
| High write load | Queue + batching   |
| Large files     | Blob storage       |
| DB overload     | Sharding           |
| High QPS        | Horizontal scaling |

### Memory hook

> Bottleneck → component

## 2.2 Latency (order of magnitude)

| Operation          | Time    |
| ------------------ | ------- |
| L1 cache           | ~1 ns   |
| RAM                | ~100 ns |
| SSD read           | ~200 µs |
| Datacenter network | ~0.5 ms |
| Cross-region       | ~70 ms  |



### Insight

Huge gaps → IO dominates cost

### Memory hook

> ns → µs → ms → network worst
