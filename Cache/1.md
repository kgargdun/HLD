---

## **1.1 What is Caching (Interview Framing)**

**Definition (say this in interview):**
Caching is storing frequently accessed data in a faster storage layer to reduce latency, improve throughput, and reduce load on downstream systems.

**Core intuition:**

* Trade **memory (cost)** for **speed (latency)**
* Avoid repeated expensive operations (DB calls, computation, network)

**Where it helps:**

* Read-heavy systems (feeds, product pages, search)
* Expensive computations (aggregation, ranking)

**Where it hurts:**

* Write-heavy systems
* Strong consistency systems (banking, inventory)

---

## **1.2 Why Caching Works**

### Temporal Locality

* Same data accessed repeatedly in short time
* Example: trending tweets, viral posts

### Spatial Locality

* Related data accessed together
* Example: user profile + posts

### Hot vs Cold Data

* **Hot data** → frequently accessed → ideal for cache
* **Cold data** → rarely accessed → keep in DB

**Interview framing:**

> “Most production traffic follows a skew (Zipf distribution), so a small % of data gets most of the reads.”

---

## **1.3 Types of Caches (System Placement)**

### 1. Client-side Cache

* Browser / mobile cache
* Reduces server calls

### 2. CDN / Edge Cache

* Close to users geographically
* Static assets, API responses

### 3. Application Cache

* Inside backend service
* Example: Redis

### 4. Database Cache

* Query cache / buffer pool
* Built into DB layer

---

## **1.4 Key Metrics (Interview Gold)**

### Cache Hit Ratio

* % of requests served from cache
* Higher = better

### Miss Penalty

* Cost of fetching from DB when cache miss happens

### Eviction Rate

* How often data is removed from cache

### Staleness Window

* How outdated cached data can be

**Interview line:**

> “A cache is only useful if hit rate is high and miss penalty is significant.”

---

## **1.5 Cacheability Heuristics**

When deciding **“should I cache this?”**

### Good Candidates

* Read-heavy endpoints
* Expensive DB queries
* Aggregated data
* Semi-static data (profiles, configs)

### Bad Candidates

* Highly dynamic data
* Low reuse data
* User-specific highly personalized data (sometimes)

### Key Properties

* **Idempotent reads** (safe to cache)
* **Immutable data → easiest**
* **Mutable data → need invalidation strategy**

---

## **1.6 When NOT to Cache**

### 1. Strong Consistency Required

* Banking balance
* Inventory during checkout

### 2. High Write Frequency

* Cache constantly invalidated → useless

### 3. Low Reuse

* Unique queries → no benefit

### 4. Large Data Objects

* Memory inefficiency

---

## **Mental Model (Most Important for HLD)**

Think of cache as:

> “A probabilistic performance optimization layer with consistency tradeoffs.”

Always ask:

1. What is my **access pattern**?
2. What is my **tolerance for stale data**?
3. What is my **expected hit rate**?
4. What happens on a **cache miss**?

---

## **How to Use This in Interviews**

When designing any system:

* Start without cache (DB baseline)
* Add cache when:

  * Latency too high
  * DB overloaded

Then explicitly mention:

* Cache location (where?)
* Cache strategy (how?)
* Invalidation (when?)
