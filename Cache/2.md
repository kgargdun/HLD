## **Cache Design Patterns (Core Interview Content).md**

---

## **2.1 Cache-Aside (Lazy Loading)**

**Most common pattern (default answer in interviews)**

### Flow

1. Application checks cache
2. If **hit → return**
3. If **miss → fetch from DB**
4. Store in cache
5. Return response

```
Client → App → Cache → (miss) → DB → Cache → Client
```

### Pros

* Simple, flexible
* Cache only what is needed
* Works with any cache (e.g., Redis)

### Cons

* Cache + DB inconsistency risk
* First request is slow (cold start)

### Failure Scenarios

* Cache miss storm → DB overload
* Stale data if not invalidated

### Interview Line

> “I’ll start with cache-aside since it gives control and is widely used in production systems.”

---

## **2.2 Read-Through Cache**

**Cache sits between app and DB**

### Flow

1. App queries cache
2. Cache fetches from DB on miss automatically

```
Client → App → Cache → DB
```

### Pros

* Simpler application logic
* Centralized cache handling

### Cons

* Less control
* Harder to debug
* Cache layer becomes complex

### Real-world Note

* Less common in custom systems
* Seen in managed caching layers

---

## **2.3 Write-Through Cache**

**Write goes to cache + DB synchronously**

### Flow

1. Write to cache
2. Cache writes to DB

```
Client → App → Cache → DB
```

### Pros

* Cache always consistent with DB
* Reads always fast (warm cache)

### Cons

* Higher write latency
* Cache stores unnecessary data

### When to Use

* Read-heavy + consistency needed

---

## **2.4 Write-Back (Write-Behind)**

**Write to cache first, DB later (async)**

### Flow

1. Write to cache
2. Async write to DB

```
Client → App → Cache → (async) → DB
```

### Pros

* Very fast writes
* Reduced DB load

### Cons (IMPORTANT)

* Data loss risk (cache crash before DB write)
* Complex recovery

### When to Use

* High write throughput systems (logs, analytics)

---

## **2.5 Refresh-Ahead / Preloading**

**Refresh cache before expiry**

### Flow

* Cache detects upcoming expiration
* Refreshes in background

### Pros

* Reduces cache misses
* Smooth latency

### Cons

* Extra compute cost
* May refresh unused data

### Use Cases

* Predictable traffic (e.g., trending content)

---

## **2.6 Hybrid Patterns (Real Systems)**

Most real systems combine patterns:

### Common Combos

#### Cache-Aside + TTL

* Simple + widely used
* Accepts slight staleness

#### Cache-Aside + Write-Through

* Reads flexible, writes consistent

#### Cache + Async Invalidation

* Event-driven updates

---

## **Comparison Summary (Important for Interviews)**

| Pattern       | Read Latency | Write Latency | Consistency | Complexity |
| ------------- | ------------ | ------------- | ----------- | ---------- |
| Cache-Aside   | Fast (hit)   | Medium        | Medium      | Low        |
| Read-Through  | Fast         | Medium        | Medium      | Medium     |
| Write-Through | Fast         | High          | Strong      | Medium     |
| Write-Back    | Fast         | Very Low      | Weak        | High       |

---

## **How to Choose (Interview Heuristic)**

### Default Answer

→ Start with **Cache-Aside**

### Then adapt:

* Need strong consistency → Write-Through
* Need high write throughput → Write-Back
* Want abstraction → Read-Through

---

## **Interview Playbook (Critical)**

When interviewer asks:

> “How would you add caching?”

Answer structure:

1. “I’ll use cache-aside pattern”
2. “On read: check cache → fallback to DB”
3. “On write: update DB + invalidate cache”
4. “Add TTL to avoid stale data buildup”

Then expect follow-ups:

* “What about cache stampede?”
* “What about stale data?”
* “What if cache fails?”

---

## **Common Mistakes**

* Not mentioning **invalidation**
* Using write-back without acknowledging **data loss risk**
* Blindly caching everything
* Ignoring **cold start problem**
