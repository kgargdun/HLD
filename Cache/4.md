## **Distributed Caching & Scaling.md**

---

## **4.1 Single Node vs Distributed Cache**

### Single Node Cache

**Setup:**

* One cache instance (e.g., Redis)

**Pros**

* Simple
* Strong consistency (single source of truth)

**Cons**

* Memory limits
* Single point of failure
* Cannot scale horizontally

---

### Distributed Cache

**Setup:**

* Multiple cache nodes
* Data partitioned across nodes

**Pros**

* Scalable
* Fault tolerant (if designed well)

**Cons**

* Complexity
* Invalidation becomes hard
* Network overhead

---

**Interview Line:**

> “Single-node cache works until memory or QPS becomes a bottleneck, then we move to distributed caching.”

---

## **4.2 Partitioning Strategies (Sharding)**

### Key-Based Partitioning

```
node = hash(key) % N
```

**Pros**

* Simple

**Cons**

* Rebalancing is painful (massive reshuffle)

---

### Consistent Hashing (IMPORTANT)

* Hash both **nodes and keys** on a ring
* Each key maps to nearest node

**Benefits:**

* Minimal data movement when nodes added/removed

**Enhancement:**

* Virtual nodes (better load balancing)

---

**Interview Line:**

> “I’ll use consistent hashing to distribute keys and minimize rebalancing impact.”

---

## **4.3 Replication**

### Why replicate?

* Fault tolerance
* High availability

---

### Leader–Follower

* Writes → leader
* Reads → followers

**Pros**

* Simple

**Cons**

* Replication lag → stale reads

---

### Multi-Master

* Writes allowed on multiple nodes

**Pros**

* High availability

**Cons**

* Conflict resolution needed

---

**Interview Note:**

* Most caches prioritize **availability over strict consistency**

---

## **4.4 Cache Coherency**

### Problem

* Same key exists in multiple nodes
* Updates → inconsistency

---

### Approaches

#### 1. Strong Coherency

* Synchronous invalidation across nodes

**Cons:** slow, not scalable

---

#### 2. Eventual Coherency (Common)

* Async updates
* Temporary inconsistency allowed

---

#### 3. Gossip-Based Sync (Advanced)

* Nodes share state periodically

---

**Interview Line:**

> “Most large-scale systems accept eventual consistency in caching for better performance.”

---

## **4.5 Handling Failures**

### Node Failure

**What happens?**

* Data in that node is lost (if no replication)

---

### Strategies

#### 1. Replication

* Backup copies exist

---

#### 2. Fallback to DB

* Cache miss → DB

---

#### 3. Graceful Degradation

* Serve stale data if possible

---

### Cache Cluster Failure

* Entire cache unavailable

**Approach:**

* System should still work (DB fallback)
* Add rate limiting to protect DB

---

**Interview Insight:**

> “Cache should never be a single point of failure — system must degrade gracefully.”

---

## **4.6 Rebalancing & Hotspots**

### Rebalancing

When nodes added/removed:

* Data redistributed

**Problem:**

* Sudden spike in cache misses

---

**Solution:**

* Consistent hashing
* Gradual warm-up

---

### Hotspots (VERY IMPORTANT)

**Problem:**

* Some keys receive massive traffic

**Example:**

* Viral post
* Celebrity profile

---

### Solutions

#### 1. Replicate Hot Keys

* Store same key in multiple nodes

---

#### 2. Request Load Balancing

* Distribute reads across replicas

---

#### 3. Local Cache Layer

* Per-instance caching

---

#### 4. Rate Limiting

* Protect backend

---

**Interview Line:**

> “Hot keys can overload a single shard, so I’d replicate or distribute reads across nodes.”

---

## **Architecture Patterns (Important)**

### 1. Client-side Sharding

* Application decides which node

**Pros**

* No extra hop

**Cons**

* Complex logic in app

---

### 2. Proxy Layer

* App → proxy → cache nodes

**Pros**

* Centralized logic

**Cons**

* Extra latency

---

### 3. Managed Cache

* Cloud-managed (abstracts complexity)

---

## **Tradeoffs Summary**

| Dimension   | Option A    | Option B    |
| ----------- | ----------- | ----------- |
| Simplicity  | Single node | Distributed |
| Consistency | Strong      | Eventual    |
| Scalability | Limited     | High        |
| Complexity  | Low         | High        |

---

## **Interview Playbook**

When scaling cache:

1. Start with:

   * Single node (Redis)

2. Scale:

   * Add sharding (consistent hashing)

3. Add:

   * Replication for HA

4. Handle:

   * Hot keys
   * Failures

---

## **Common Mistakes**

* Ignoring rebalancing cost
* Not planning for node failure
* Assuming cache is always available
* Ignoring hotspot issues

---

## **Golden Rule**

> “Distributed caching is easy to add, but hard to keep correct.”
