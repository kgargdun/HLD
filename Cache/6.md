## **Pitfalls, Tradeoffs & Interview Playbook.md**

---

## **6.1 Classic Problems (Must Know)**

### 1. Cache Stampede (Thundering Herd)

**Problem:**

* Cache expires → many requests hit DB simultaneously

**Impact:**

* DB overload → cascading failure

---

### 2. Cache Penetration

**Problem:**

* Requests for non-existent keys always miss cache

**Example:**

* Invalid user IDs

---

### 3. Cache Avalanche

**Problem:**

* Many keys expire at the same time

**Impact:**

* Sudden DB spike

---

### 4. Hot Key Problem

**Problem:**

* One key gets massive traffic

**Impact:**

* Single shard/node overloaded

---

## **6.2 Mitigation Techniques**

### Cache Stampede Solutions

* **Request Coalescing**

  * Only one request fetches DB, others wait

* **Soft TTL**

  * Serve stale data while refreshing

* **Jittered TTL**

  * Add randomness to expiry

---

### Cache Penetration Solutions

* **Cache null values**
* **Bloom filters** (prevent invalid queries reaching DB)

---

### Cache Avalanche Solutions

* Randomized TTLs
* Gradual expiration
* Pre-warming cache

---

### Hot Key Solutions

* Replicate hot keys
* Local in-memory caching
* Load balancing across replicas

---

## **6.3 Tradeoffs (This is What Interviewers Care About)**

### Consistency vs Performance

* Strong consistency → slower
* Eventual consistency → faster

---

### Freshness vs Latency

* Fresh data → more DB hits
* Cached data → lower latency but stale

---

### Cost vs Hit Rate

* More memory → higher hit rate
* Less memory → more misses

---

### Complexity vs Control

* Simple TTL → easy but less control
* Event-driven invalidation → powerful but complex

---

**Interview Line:**

> “Caching is always a tradeoff between consistency, latency, and cost.”

---

## **6.4 Observability (Often Ignored, High Signal)**

### Metrics to Track

* Cache hit ratio
* Miss rate
* Latency (cache vs DB)
* Eviction rate

---

### Alerts

* Sudden drop in hit rate
* Spike in DB traffic
* Cache node failures

---

### Debugging Strategy

* Check hit/miss patterns
* Validate key design
* Look for uneven distribution (hot keys)

---

## **6.5 Security & Edge Cases**

### Sensitive Data in Cache

* Avoid caching:

  * Passwords
  * Tokens (unless encrypted)

---

### Multi-Tenant Systems

* Key namespacing required

---

### Cache Poisoning

* Malicious data stored in cache

---

### Large Objects

* Memory inefficiency
* Consider partial caching

---

## **6.6 Interview Playbook (Very Important)**

### When to Introduce Cache

* When:

  * DB latency high
  * Read QPS high

---

### How to Introduce (Step-by-Step Answer)

1. “Start with DB-backed system”
2. “Add Redis cache using cache-aside pattern”
3. “On read → check cache → fallback to DB”
4. “On write → update DB + invalidate cache”
5. “Add TTL for safety”
6. “Handle edge cases like stampede”

---

### Expected Follow-ups

Be ready for:

* “What if cache fails?”
* “How do you handle stale data?”
* “How do you scale cache?”
* “What about hot keys?”

---

### Strong Closing Line

> “Cache improves performance, but I’ll design the system to remain correct even without it.”

---

## **Common Mistakes (Reject Signals)**

* Treating cache as source of truth
* Ignoring invalidation
* No failure handling
* Over-engineering early

---

## **Final Mental Model**

> “Cache is an optimization layer — not part of correctness.”

---

## **How to Stand Out**

Most candidates:

* Say “use Redis”

Strong candidates:

* Explain pattern + invalidation + tradeoffs

Top candidates:

* Mention failure modes + mitigation + metrics
