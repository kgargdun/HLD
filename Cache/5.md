## **5.1 In-Memory Caching Systems**

### Redis

**What it is:**

* In-memory key-value store
* Supports data structures (strings, lists, sets, sorted sets)

**Why it's popular:**

* Extremely fast (RAM-based)
* Rich features (TTL, pub/sub, persistence)

**Used for:**

* API caching
* Session storage
* Leaderboards
* Rate limiting

---

### Memcached

**What it is:**

* Simple distributed in-memory cache

**Pros**

* Very lightweight
* Easy to scale

**Cons**

* No persistence
* Limited features (compared to Redis)

---

**Interview Insight:**

> “Redis is feature-rich, Memcached is simpler and faster for pure caching.”

## **5.2 CDN / Edge Caching**

### Cloudflare

### Akamai

**What they do:**

* Cache content at edge locations near users

**Used for:**

* Static assets (images, JS, CSS)
* API response caching (in some cases)

---

**Benefits:**

* Reduces latency (geo proximity)

---

**Interview Use Case:**

> “For global systems, I’d add CDN caching to reduce latency and backend load.”

## **5.3 Application-Level Caching**

### Where it lives:

* Inside your service layer

---

### Examples:

* In-memory maps (per instance)
* LRU caches inside app

---

### When to use:

* Extremely hot data
* Low latency requirement

---

**Problem:**

* Each instance has its own cache → inconsistency

---

**Solution:**

* Combine with distributed cache

---

**Interview Line:**

> “I’d use local cache for ultra-hot data and Redis for shared consistency.”

---

## **5.4 Database-Level Caching**

### Built-in DB caching

#### MySQL (InnoDB buffer pool)

* Caches frequently accessed pages

#### PostgreSQL shared buffers

* Similar concept

---

### Query Cache (less common now)

* Stores query results

---

**Key Insight:**

> “Even without explicit caching, databases already cache data internally.”

---

### When to rely on DB cache:

* Simple systems
* Low scale

---

### When NOT enough:

* High QPS systems → need external cache (Redis)

## **5.5 Cache in Real Systems**

### 1. Feed Systems (Instagram, Twitter)

**What is cached:**

* User feed
* Post metadata

**Pattern:**

* Cache-aside + TTL
* Precompute feeds

---

### 2. Search Systems

**What is cached:**

* Query results
* Trending searches

**Challenge:**

* High cardinality (many unique queries)

---

### 3. Rate Limiting

**Tool:**

* Redis

**Approach:**

* Counter per user/IP
* TTL-based window

---

### 4. Session Storage

* Store session tokens in Redis
* Fast access + expiry

---

### 5. Product Catalog (E-commerce)

* Cache product details
* Invalidate on updates

---

## **5.6 Decision Framework (Very Important)**

When choosing caching approach:

---

### Step 1: What to Cache?

* Read-heavy data
* Expensive queries

---

### Step 2: Where to Cache?

| Layer     | When to Use            |
| --------- | ---------------------- |
| CDN       | Static/global content  |
| App cache | Ultra-hot, low latency |
| Redis     | Shared cache           |
| DB cache  | Small/simple systems   |

---

### Step 3: Pattern?

* Default → Cache-aside
* High write consistency → Write-through
* High write throughput → Write-back

---

### Step 4: TTL?

* Based on freshness requirement

---

### Step 5: Invalidation?

* Write invalidate (default)
* Event-driven (scale)

---
