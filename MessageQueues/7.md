# Message Queues — Real World Mapping (Interview Gold)

> This file connects **theory → actual systems**

---

## 1. Apache Kafka — Event Streaming Backbone

### What it is best at

* High-throughput event streaming
* Log aggregation
* Analytics pipelines

---

### Core Model

* Partitioned, append-only log
* Pull-based consumption
* Retention-based storage

---

### Typical Use Cases

* Activity tracking (clickstreams)
* Metrics pipelines
* Event-driven architectures

---

### When to choose Kafka

> Use Kafka when you need **scale, replay, and streaming**

---

## 2. RabbitMQ — Traditional Message Broker

### What it is best at

* Task queues
* Complex routing (exchanges)
* Low-latency messaging

---

### Core Model

* Queue-based
* Push-based delivery
* Messages deleted after ACK

---

### Strengths

* Flexible routing (fanout, topic, direct)
* Mature ecosystem

---

### When to choose RabbitMQ

> Use RabbitMQ for **task processing and complex routing**

---

## 3. Amazon SQS — Fully Managed Queue

### What it is best at

* Simple, scalable queues
* Serverless architectures

---

### Features

* At-least-once delivery
* Visibility timeout
* Dead Letter Queues

---

### Types

#### (a) Standard Queue

* High throughput
* Best-effort ordering

#### (b) FIFO Queue

* Ordering guaranteed
* Lower throughput

---

### When to choose SQS

> Use SQS when you want **managed infrastructure with minimal ops**

---

## 4. Google Pub/Sub — Cloud Pub/Sub System

### What it is best at

* Event-driven systems
* Global scale

---

### Features

* Pub/Sub model
* Auto scaling
* At-least-once delivery

---

### When to choose

> Use when building **event-driven systems on GCP**

---

## 5. Quick Comparison (Interview Ready)

| Feature    | Kafka         | RabbitMQ     | SQS         | Pub/Sub |
| ---------- | ------------- | ------------ | ----------- | ------- |
| Model      | Log (stream)  | Queue        | Queue       | Pub/Sub |
| Throughput | Very High     | Medium       | High        | High    |
| Ordering   | Per partition | Queue-level  | FIFO option | Limited |
| Replay     | ✅ Yes         | ❌ No         | ❌ No        | ❌ No    |
| Management | Self-managed  | Self-managed | Managed     | Managed |

---

## 6. Real Production Architecture ⭐

### Multi-System Usage

```id="rw3k9p"
User Action
   ↓
Kafka → Event Streaming / Analytics
   ↓
SQS / RabbitMQ → Background Jobs
   ↓
Workers → Processing
```

---

### Why multiple systems?

* Kafka → streaming + durability
* SQS/RabbitMQ → task execution

---

## 7. Real Company Patterns

### (a) E-commerce

* Orders → Queue → Workers
* Events → Kafka → Analytics

---

### (b) Social Media

* User activity → Kafka
* Notifications → Queue

---

### (c) Microservices

* Event-driven via Kafka / PubSub
* Async jobs via SQS

---

## 8. Decision Framework (VERY IMPORTANT)

### Ask these in interview:

#### 1. Throughput needed?

* High → Kafka

#### 2. Need replay?

* Yes → Kafka

#### 3. Complex routing?

* Yes → RabbitMQ

#### 4. Want managed service?

* Yes → SQS / PubSub

---

## 9. What to Say in Interviews

### Strong Answer

> “For high-throughput event streaming, I’d use Kafka. For background job processing, I’d use SQS or RabbitMQ. In production, systems often combine both — Kafka for streaming and queues for task execution.”
