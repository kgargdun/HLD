# Message Queues — Design Patterns (Interview Focus)

> This file answers:
> **How do we actually use message queues in real systems?**

---

## 1. Work Queue (Task Queue)

### Idea

* Multiple consumers process tasks from a queue
* Each message is handled by **only one consumer**

---

### Flow

```id="wq1x9p"
Producer → Queue → Consumer Pool
```

---

### Example

* Image processing
* Email sending
* Order fulfillment

---

### Key Benefit

* Parallel processing → scalability

---

### Interview Insight

> “Work queues are used for background job processing with horizontal scaling via multiple consumers.”

---

## 2. Publish–Subscribe (Pub/Sub)

### Idea

* One message → multiple consumers
* Each consumer gets a copy

---

### Flow

```id="ps8k2m"
Producer → Topic → Multiple Subscribers
```

---

### Example

* User signup event:

  * Email service
  * Analytics service
  * Recommendation system

---

### Systems

* Apache Kafka
* Google Pub/Sub

---

### Interview Insight

> “Pub/Sub enables event-driven architectures where multiple services react to the same event independently.”

---

## 3. Fan-Out Pattern

### Idea

* One message → replicated into multiple queues

---

### Difference from Pub/Sub

* Each queue is independent
* Consumers don’t interfere

---

### Flow

```id="f3n7op"
Producer → Exchange → Queue A
                     → Queue B
                     → Queue C
```

---

### System Example

* RabbitMQ (exchange-based routing)

---

### Use Case

* Different teams/services consuming same event

---

## 4. Dead Letter Queue (DLQ) ⭐

### Idea

* Failed messages are moved to a separate queue

---

### Why needed?

* Prevent infinite retries
* Debug failures

---

### Flow

```id="dlq9z1"
Main Queue → Consumer → FAIL → DLQ
```

---

### Use Cases

* Poison messages
* Schema mismatch
* Persistent failures

---

### Interview Insight

> “DLQ is essential for reliability and debugging in production systems.”

---

## 5. Retry Pattern

### Problem

* Transient failures (network, DB down)

---

### Solution

* Retry message processing

---

### Strategies

#### (a) Immediate Retry

* Retry instantly (simple but risky)

---

#### (b) Delayed Retry (Better)

```id="rt2m8x"
Retry after 1s → 5s → 30s → ...
```

---

#### (c) Exponential Backoff ⭐

* Increasing delay between retries

---

### Why important?

* Prevents system overload

---

## 6. Delayed / Scheduled Messages

### Idea

* Process message after a delay

---

### Example

* Send reminder after 24 hours
* Retry after delay

---

### Systems

* Amazon SQS (delay queues)

---

## 7. Priority Queue

### Idea

* Some messages processed before others

---

### Example

* VIP users > normal users

---

### Tradeoff

* Can affect fairness

---

## 8. Request–Reply Pattern

### Idea

* Simulate synchronous communication using MQ

---

### Flow

```id="rr4t1k"
Client → Request Queue → Service
       ← Response Queue ←
```

---

### Use Case

* Legacy systems
* Cross-service communication

---

### Drawback

* Adds complexity
* Not truly async

---

## 9. Competing Consumers Pattern

### Idea

* Multiple consumers compete for messages

---

### Benefit

* Load balancing
* High throughput

---

### Example

* Worker pool consuming jobs

---

## 10. Event Sourcing (Advanced)

### Idea

* Store events instead of current state

---

### Flow

```id="es7y2n"
Event → Queue → Stored as log → Rebuild state
```

---

### Systems

* Apache Kafka

---

### Use Case

* Audit logs
* Financial systems

---

## 11. Common Pitfalls

* Infinite retries → system collapse
* No DLQ → debugging nightmare
* Tight coupling via message schema
* Ignoring idempotency

---

## 12. What to Say in Interviews

> “I would use a work queue for background processing, pub/sub for event-driven communication, and include DLQs with retry and exponential backoff for reliability.”

---

## 13. One-Line Summary

> MQ design patterns like work queues, pub/sub, retries, and DLQs enable scalable, resilient, and decoupled system architectures.
