# Message Queues — Basics (Interview Focus)

> This file builds the foundation.
> Everything else (Kafka, delivery semantics, scaling) depends on this.

---

## 1. What is a Message Queue?

A **Message Queue (MQ)** is a system that enables **asynchronous communication** between components by passing messages through a queue.

---

### Core Idea

```id="a1q9mz"
Producer → Queue (Broker) → Consumer
```

* Producer sends message
* Queue stores it
* Consumer processes it later

---

## 2. Why Do We Need Message Queues?

### (a) Decoupling

* Producer and consumer don’t need to know about each other
* Can evolve independently

---

### (b) Asynchronous Processing

* Producer doesn’t wait for processing
* Improves latency

---

### (c) Load Buffering (VERY IMPORTANT)

* Absorb traffic spikes
* Smooth out load

👉 Example:

* 10k requests/sec incoming
* DB can handle 2k/sec
  → Queue buffers the rest

---

### (d) Reliability

* Messages can be retried
* Failures don’t lose data (if persisted)

---

## 3. Queue vs Pub/Sub

### Queue (Point-to-Point)

* One message → one consumer
* Work distribution

---

### Pub/Sub (Publish–Subscribe)

* One message → multiple consumers
* Each subscriber gets a copy

---

### Summary

| Feature   | Queue           | Pub/Sub            |
| --------- | --------------- | ------------------ |
| Consumers | One             | Many               |
| Use case  | Task processing | Event broadcasting |

---

## 4. Core Components

### (a) Producer

* Sends messages

---

### (b) Consumer

* Processes messages

---

### (c) Broker

* Stores and manages messages
* Examples:

  * Apache Kafka
  * RabbitMQ

---

### (d) Message

* Unit of data
* Usually:

  * JSON / binary payload
  * Metadata (timestamp, id)

---

## 5. Push vs Pull Model

### Push Model

* Broker pushes messages to consumers
* Example: RabbitMQ

---

### Pull Model

* Consumer pulls messages when ready
* Example: Apache Kafka

---

### Tradeoff

| Push          | Pull                |
| ------------- | ------------------- |
| Simpler       | More control        |
| Can overwhelm | Better backpressure |

---

## 6. Message Lifecycle

```id="k2pl0s"
Producer → Broker → Stored → Delivered → Acknowledged → Deleted
```

---

### Important Concept: ACK (Acknowledgment)

* Consumer confirms processing
* Without ACK → message may be retried

---

## 7. Key Properties of MQ Systems

### (a) Durability

* Messages survive crashes (disk storage)

---

### (b) Ordering

* Messages processed in order (not always guaranteed)

---

### (c) Delivery Guarantees

* At-most-once
* At-least-once
* Exactly-once (covered later)

---

### (d) Scalability

* Can handle large volumes via partitioning

---

## 8. When to Use Message Queues

### Common Use Cases

* Background jobs (email, notifications)
* Order processing
* Log/event pipelines
* Microservices communication

---

### Example

```id="j9w0az"
User places order
  ↓
Order Service → Queue
  ↓
Payment Service (async)
  ↓
Notification Service (async)
```

---

## 9. When NOT to Use MQ

* Real-time synchronous requirements
* Strong consistency required immediately
* Simple systems (overkill)

---

## 10. Common Pitfalls

* Ignoring retries → message loss
* Not handling duplicates
* Assuming ordering globally
* Infinite retries → system overload

---

## 11. What to Say in Interviews

> “Message queues enable asynchronous, decoupled communication between services, allowing systems to handle spikes, improve reliability, and scale independently.”

---

## 12. One-Line Summary

> Message queues decouple producers and consumers using asynchronous communication, improving scalability, reliability, and system resilience.
