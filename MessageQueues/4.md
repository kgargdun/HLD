# Message Queues — Storage & Ordering (Interview Focus)

> This file answers:
> **How messages are stored, retrieved, and ordered in real systems**

---

## 1. Message Storage

### (a) In-Memory Storage

* Messages stored in RAM

---

### Pros

* Very fast
* Low latency

### Cons

* Data loss on crash ❗
* Not durable

---

### Use Cases

* Low-critical systems
* Temporary buffering

---

## 2. Persistent Storage (Disk-Based)

Used by systems like:

* Apache Kafka
* RabbitMQ

---

### Idea

* Messages written to disk (log/file)

---

### Pros

* Durable (survives crashes)
* Reliable

### Cons

* Higher latency than memory

---

### Interview Insight

> “Most production MQ systems persist messages to disk for durability.”

---

## 3. Log-Based Storage (Kafka Model) ⭐

### Idea

* Messages appended to an **immutable log**

---

### Flow

```id="lg9p3k"
Append → Read sequentially → Never modify
```

---

### Key Properties

* Sequential writes → very fast
* No random updates
* Consumers track their own position (offset)

---

### Important Concept

> Messages are **not deleted immediately** after consumption

---

## 4. Retention vs Deletion

### Traditional MQ

* Message deleted after ACK

---

### Kafka Model

* Message retained for time/size

---

### Example

* Keep data for 7 days

---

### Benefits

* Replay messages
* Debugging
* Event sourcing

---

## 5. Ordering Guarantees

### (a) Global Ordering ❌ (Rare)

* All messages strictly ordered
* Very hard to scale

---

### (b) Partition-Level Ordering ⭐

* Order guaranteed **within a partition**

---

### Example

```id="ord8q2"
Partition 1: A → B → C (ordered)
Partition 2: X → Y → Z (ordered)
```

---

👉 No ordering across partitions

---

## 6. Partitioning

### Why needed?

* Scale horizontally
* Increase throughput

---

### How it works

```id="pt5k1m"
Topic → Partition 1
      → Partition 2
      → Partition 3
```

---

### Message Routing

* Hash(key) → partition

---

### Key Insight

> Same key → same partition → ordering preserved

---

## 7. Offsets (VERY IMPORTANT)

### Definition

* Position of message in a partition

---

### Example

```id="of2p9z"
Offset: 0, 1, 2, 3, ...
```

---

### Key Idea

* Consumer tracks offset
* Broker doesn’t track consumption state (Kafka)

---

### Benefits

* Replay capability
* Fault tolerance

---

## 8. Acknowledgment Models

### (a) Auto-ACK

* Message marked done immediately

---

### (b) Manual ACK

* Consumer explicitly acknowledges

---

### Tradeoff

| Type   | Reliability | Simplicity |
| ------ | ----------- | ---------- |
| Auto   | Low         | High       |
| Manual | High        | Medium     |

---

## 9. Ordering vs Throughput Tradeoff

### Strict Ordering

* Single partition
* Low throughput

---

### High Throughput

* Multiple partitions
* Lose global ordering

---

### Interview Insight

> “Ordering is typically guaranteed per partition, not globally, to allow horizontal scaling.”

---

## 10. Consumer Models

### (a) Push-Based

* Broker sends messages

---

### (b) Pull-Based (Kafka)

* Consumer fetches messages

---

### Benefit of Pull

* Consumer controls rate
* Better backpressure handling

---

## 11. Replay Capability (Kafka Advantage)

### Idea

* Re-read old messages

---

### Use Cases

* Debugging
* Reprocessing data
* New consumers

---

## 12. Common Pitfalls

* Assuming global ordering
* Not handling out-of-order messages
* Large messages → performance issues
* Ignoring retention limits

---

## 13. What to Say in Interviews

> “I would use a log-based storage model with partitioning for scalability, ensuring ordering within partitions and allowing consumers to track offsets for fault tolerance and replay.”

---

## 14. One-Line Summary

> MQ systems store messages either in memory or disk, with modern systems like Kafka using log-based storage, partitioning, and offsets to balance durability, ordering, and scalability.
